> **SSL (Secure Sockets Layer)** — протокол шифрования, разработанный Netscape в 1990-х. Сегодня технически **устарел и уязвим** — все его версии (SSL 2.0, 3.0) официально запрещены (RFC 7568). На практике везде используется **TLS (Transport Layer Security)** — его прямой наследник, но название «SSL» прижилось и используется как синоним в быту.

**Текущие версии:** TLS 1.2 (всё ещё распространён), TLS 1.3 (рекомендуется)  
**Где работает:** между TCP и прикладным уровнем (HTTPS, SMTPS, IMAPS, SSH не использует TLS — у него своя криптография)
```
HTTP / SMTP / FTP / ...   ← прикладной уровень
─────────────────────────
TLS                       ← шифрование здесь
─────────────────────────
TCP                       ← надёжная доставка
─────────────────────────
IP → Ethernet → физика
```

# 1. Зачем нужен TLS — три задачи

| Задача                 | Что это значит                                          |
| ---------------------- | ------------------------------------------------------- |
| **Конфиденциальность** | Данные зашифрованы — перехватить и прочитать нельзя     |
| **Целостность**        | Данные не изменены в пути — любая правка обнаруживается |
| **Аутентификация**     | Ты точно разговариваешь с тем сервером, а не с подменой |
___
# 2. Два вида шифрования
TLS использует оба вида — каждый на своём месте.
## 2.1. Асимметричное (публичный + приватный ключ)
Два математически связанных ключа: **зашифровать** можно публичным, **расшифровать** — только приватным (и наоборот для подписи).
```
Открытый текст ──[публичный ключ]──► Шифротекст
Шифротекст     ──[приватный ключ]──► Открытый текст
```
- Медленное (в 100–1000 раз медленнее симметричного)
- Не требует предварительного обмена секретом
- **Используется в TLS:** только на этапе handshake — для обмена ключами и аутентификации сервера
- Алгоритмы: **RSA**, **ECDH**, **Ed25519**
## 2.2. Симметричное (один общий ключ)
Один ключ — и для шифрования, и для расшифровки. Обе стороны должны знать этот ключ заранее.
```
Открытый текст ──[общий ключ]──► Шифротекст
Шифротекст     ──[общий ключ]──► Открытый текст
```
- Быстрое — подходит для потоков данных
- Проблема: как безопасно передать ключ? → решается асимметричным шифрованием
- **Используется в TLS:** для шифрования всех данных сессии
- Алгоритмы: **AES**, **ChaCha20**
> Комбинация такая: асимметрика договаривается об общем ключе → дальше всё шифруется симметрично. Быстро и безопасно.

# 3. Алгоритмы — краткий справочник

## 3.1. Обмен ключами (Key Exchange)
Позволяет двум сторонам выработать общий секрет через открытый канал, не передавая сам секрет.

| Алгоритм | Описание |
|----------|----------|
| **ECDH** (Elliptic Curve Diffie-Hellman) | Рекомендуется. Короткие ключи, высокая стойкость |
| **DHE** (Diffie-Hellman Ephemeral) | Классический DH с временными ключами |
| **RSA** (обмен ключами) | Устарел в TLS 1.3 — нет forward secrecy |
## 3.2. Аутентификация сервера (подпись сертификата)

| Алгоритм | Описание |
|----------|----------|
| **RSA** | Самый распространённый. Ключ 2048–4096 бит |
| **ECDSA** | Эллиптические кривые, короче и быстрее RSA |
| **Ed25519** | Современный, рекомендуется |
## 3.3. Симметричное шифрование (данные сессии)

| Алгоритм | Описание |
|----------|----------|
| **AES-128-GCM** | Рекомендуется. Шифрование + аутентификация вместе (AEAD) |
| **AES-256-GCM** | То же, длиннее ключ |
| **ChaCha20-Poly1305** | Быстрый без аппаратного ускорения AES (мобильные устройства) |
| **AES-CBC** | Устарел, уязвим (BEAST, POODLE атаки) |
## 3.4. Хэширование / целостность (MAC)

| Алгоритм | Описание |
|----------|----------|
| **SHA-256** | Рекомендуется |
| **SHA-384** | Усиленный вариант |
| **SHA-1** | Устарел, сломан |
| **MD5** | Устарел, сломан |
___
# 4. Cipher Suite — набор алгоритмов
**Cipher Suite** — строка, описывающая полный набор алгоритмов для одной TLS-сессии. Клиент и сервер договариваются, какой набор использовать.
Формат (TLS 1.2):
```
TLS _ [Обмен ключами] _ [Аутентификация] _ WITH _ [Шифрование] _ [MAC]
```
Примеры:
```
TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256     ← хороший, распространённый
TLS_ECDHE_ECDSA_WITH_CHACHA20_POLY1305    ← рекомендуется
TLS_RSA_WITH_AES_256_CBC_SHA              ← устарел (нет forward secrecy)
TLS_RSA_WITH_RC4_128_MD5                  ← сломан, запрещён
```
В **TLS 1.3** формат упрощён — обмен ключами вынесен отдельно, в cipher suite только шифрование и хэш:
```
TLS_AES_128_GCM_SHA256         ← стандарт TLS 1.3
TLS_AES_256_GCM_SHA384
TLS_CHACHA20_POLY1305_SHA256
```

# 5. Сертификаты и цепочка доверия
> Асимметричное шифрование решает задачу конфиденциальности, но не аутентификации. Кто угодно может создать пару ключей. **Сертификат** — это публичный ключ, подписанный доверенной третьей стороной.
## 5.1. Что такое X.509 сертификат
Файл, содержащий:
- Публичный ключ сервера
- Домен (Common Name / SAN)
- Срок действия
- Подпись удостоверяющего центра (CA — Certificate Authority)
## 5.2. Цепочка доверия
```
Root CA (корневой — встроен в ОС/браузер)
    │  подписывает
    ▼
Intermediate CA (промежуточный)
    │  подписывает
    ▼
Сертификат сайта (example.com)
```
Браузер доверяет ~150 корневым CA. Если цепочка подписей приводит к одному из них — сайту можно доверять.
## 5.3. Типы сертификатов

| Тип | Что проверяет CA | Где виден |
|-----|-----------------|-----------|
| **DV** (Domain Validation) | Только владение доменом | Замочек в браузере |
| **OV** (Organization Validation) | Домен + организация | Детали сертификата |
| **EV** (Extended Validation) | Полная проверка компании | Зелёная строка (устарело) |
| **Wildcard** `*.example.com` | Все поддомены | — |
| **Self-signed** | Никто (сам себя подписал) | Предупреждение браузера |
**Let's Encrypt** — бесплатный CA, автоматически выдаёт DV-сертификаты. Де-факто стандарт для большинства сайтов.

# 6. TLS Handshake

## 6.1. TLS 1.2
```
Клиент                                        Сервер
  │                                               │
  │──── ClientHello ───────────────────────────►  │
  │  версия TLS, случайное число,                 │
  │  список поддерживаемых cipher suites          │
  │                                               │
  │◄─── ServerHello ───────────────────────────   │
  │  выбранный cipher suite, случайное число      │
  │                                               │
  │◄─── Certificate ───────────────────────────   │
  │  сертификат сервера (публичный ключ)          │
  │                                               │
  │◄─── ServerHelloDone ───────────────────────   │
  │                                               │
  │  Клиент проверяет сертификат                  │
  │  (цепочка доверия, срок, домен)               │
  │                                               │
  │──── ClientKeyExchange ──────────────────────► │
  │  pre-master secret (зашифрован pub ключом)    │
  │                                               │
  │  Обе стороны вычисляют master secret          │
  │  → сессионные ключи шифрования                │
  │                                               │
  │──── ChangeCipherSpec + Finished ────────────► │
  │◄─── ChangeCipherSpec + Finished ────────────  │
  │                                               │
  │  [ весь трафик зашифрован ]                   │
```
**Проблема TLS 1.2:** 2 RTT до начала передачи данных. Плюс некоторые cipher suites без DHE/ECDHE не имеют **forward secrecy**.
## 6.2. TLS 1.3 — что изменилось
```
Клиент                                        Сервер
  │                                               │
  │──── ClientHello ───────────────────────────►  │
  │  + ключи для обмена (key_share)               │  ← сразу отправляет DH-параметры
  │                                               │
  │◄─── ServerHello + Certificate + Finished ───  │  ← всё в одном ответе
  │                                               │
  │──── Finished ───────────────────────────────► │
  │                                               │
  │  [ трафик зашифрован ]                        │
```

| | TLS 1.2 | TLS 1.3 |
|---|---------|---------|
| Handshake | 2 RTT | **1 RTT** |
| 0-RTT (повторное подключение) | Нет | Есть (с оговорками) |
| Forward Secrecy | Опционально | **Всегда** |
| Устаревшие алгоритмы | Допускаются | **Запрещены** (RC4, MD5, SHA-1, RSA key exchange) |
| Шифрование handshake | Частичное | **Полное** (Certificate тоже зашифрован) |
___
# 7. Forward Secrecy — почему важно
> **Forward Secrecy (PFS)** — свойство: компрометация долгосрочного приватного ключа сервера **не раскрывает** прошлые сессии.
**Без PFS (RSA key exchange):**
```
Злоумышленник записал зашифрованный трафик за год.
Через год украл приватный ключ сервера.
→ Расшифровал весь записанный трафик. ✗
```
**С PFS (ECDHE):**
```
Для каждой сессии генерируются временные (ephemeral) ключи.
После сессии они уничтожаются.
Кража долгосрочного ключа не поможет расшифровать прошлое. ✓
```
В TLS 1.3 forward secrecy обязателен — RSA key exchange убрали из стандарта.

# 8. HTTPS = HTTP + TLS
```
браузер вводит https://example.com
         │
         ▼
TCP handshake (порт 443)
         │
         ▼
TLS handshake (проверка сертификата, выработка ключей)
         │
         ▼
HTTP-запрос внутри зашифрованного TLS-канала
         │
         ▼
HTTP-ответ (тоже зашифрован)
```
**HTTP Strict Transport Security (HSTS)** — заголовок, который говорит браузеру: «всегда подключайся только по HTTPS, никогда по HTTP». Защищает от downgrade-атак.
