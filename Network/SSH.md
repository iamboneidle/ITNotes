# 1. Место SSH в стеке протоколов
```
┌─────────────────────────────┐
│   Прикладной (L7)           │  SSH (команды, файлы, туннели)
├─────────────────────────────┤
│   Транспортный (L4)         │  TCP — надёжная доставка
├─────────────────────────────┤
│   Сетевой (L3)              │  IP — маршрутизация
├─────────────────────────────┤
│   Канальный (L2)            │  Ethernet, Wi-Fi
└─────────────────────────────┘
```
**Связь с TCP:** SSH требует надёжного соединения — гарантированной доставки и порядка байт. Именно поэтому под ним TCP, а не UDP. Прежде чем SSH начнёт работу, TCP уже выполнил трёхстороннее рукопожатие и установил соединение на порту 22.

# 2. Архитектура SSH — три подпротокола
SSH внутри состоит из трёх слоёв, каждый из которых RFC описывает отдельно:
```
┌──────────────────────────────────────────┐
│  SSH Connection Protocol (RFC 4254)      │  каналы, туннели, псевдотерминал
├──────────────────────────────────────────┤
│  SSH Authentication Protocol (RFC 4252)  │  аутентификация пользователя
├──────────────────────────────────────────┤
│  SSH Transport Layer Protocol (RFC 4253) │  шифрование, целостность, handshake
└──────────────────────────────────────────┘
                    │
                   TCP
```

| Подпротокол | Назначение |
|-------------|-----------|
| **Transport Layer** | Согласование алгоритмов, обмен ключами, шифрование канала |
| **Authentication** | Проверка личности пользователя (пароль, ключ, и др.) |
| **Connection** | Мультиплексирование нескольких логических каналов внутри одного TCP-соединения |

# 3. Криптография в SSH
## 3.1. Симметричное шифрование (данные сессии)
После handshake весь трафик шифруется **симметричным** алгоритмом — быстро, ключ знают обе стороны.

| Алгоритм | Описание |
|----------|----------|
| `AES-128/256-CTR` | Рекомендуется, используется чаще всего |
| `AES-128/256-GCM` | AES с аутентификацией (AEAD) |
| `ChaCha20-Poly1305` | Быстрый на устройствах без AES-инструкций |
| `3DES-CBC` | Устаревший, не рекомендуется |
## 3.2. Асимметричное шифрование (аутентификация и обмен ключами)
Используется **только на этапе handshake** и при аутентификации по ключам. Медленно, но позволяет договориться без общего секрета.

| Алгоритм | Описание |
|----------|----------|
| `Ed25519` | Рекомендуется. Быстрый, короткие ключи, стойкий |
| `ECDSA` | Эллиптические кривые |
| `RSA-4096` | Классика, всё ещё надёжен при длине ≥ 3072 бит |
| `DSA` | Устарел, отключён в современных реализациях |
## 3.3. MAC — целостность сообщений
Каждый пакет подписывается кодом аутентификации сообщения (HMAC), чтобы обнаружить подделку.

| Алгоритм | Описание |
|----------|----------|
| `hmac-sha2-256` | Рекомендуется |
| `hmac-sha2-512` | Более стойкий |
| `hmac-sha1` | Устарел |
## 3.4. Обмен ключами (Key Exchange)
Позволяет двум сторонам выработать общий симметричный ключ через **незащищённый** канал, не передавая сам ключ.

| Алгоритм | Описание |
|----------|----------|
| `curve25519-sha256` | Рекомендуется. ECDH на кривой Curve25519 |
| `ecdh-sha2-nistp256` | ECDH на кривой NIST P-256 |
| `diffie-hellman-group14-sha256` | Классический DH, 2048 бит |
| `diffie-hellman-group1-sha1` | Устарел, уязвим |

# 4. SSH Handshake — пошагово

```
Клиент                                        Сервер
  │                                               │
  │◄══════════ TCP трёхстороннее рукопожатие ════►│
  │                                               │
  │──── SSH-2.0-OpenSSH_9.x ───────────────────►  │  [1] Обмен версиями
  │◄─── SSH-2.0-OpenSSH_9.x ───────────────────   │
  │                                               │
  │──── SSH_MSG_KEXINIT ───────────────────────►  │  [2] Согласование алгоритмов
  │  (список поддерживаемых kex, cipher, mac)     │
  │◄─── SSH_MSG_KEXINIT ───────────────────────   │
  │  (выбирается пересечение списков)             │
  │                                               │
  │──── SSH_MSG_KEX_ECDH_INIT (pubkey_client) ──► │  [3] Обмен ключами (DH/ECDH)
  │◄─── SSH_MSG_KEX_ECDH_REPLY ────────────────   │
  │  (pubkey_server, host_key, signature)         │
  │                                               │
  │  Клиент проверяет host_key сервера            │  [4] Проверка подлинности сервера
  │  (known_hosts или первое подключение)         │
  │                                               │
  │  Обе стороны вычисляют общий секрет           │  [5] Выработка сессионных ключей
  │  → симметричные ключи шифрования              │
  │                                               │
  │──── SSH_MSG_NEWKEYS ────────────────────────► │  [6] Переключение на шифрование
  │◄─── SSH_MSG_NEWKEYS ────────────────────────  │
  │                                               │
  │  [ весь дальнейший трафик зашифрован ]        │
  │                                               │
  │──── SSH_MSG_USERAUTH_REQUEST ───────────────► │  [7] Аутентификация пользователя
  │◄─── SSH_MSG_USERAUTH_SUCCESS ───────────────  │
  │                                               │
  │──── SSH_MSG_CHANNEL_OPEN ───────────────────► │  [8] Открытие канала (терминал и т.д.)
  │◄─── SSH_MSG_CHANNEL_OPEN_CONFIRMATION ──────  │
  │                                               │
  │  [ работа: команды, файлы, туннели ]          │
```

> [!NOTE] Важный момент на шаге 4
> При **первом** подключении к серверу клиент не знает его host_key и спрашивает:
> `The authenticity of host '...' can't be established. Are you sure you want to continue?`
> После согласия ключ сохраняется в `~/.ssh/known_hosts`. При следующих подключениях ключ сверяется автоматически — если он изменился, SSH выдаст предупреждение (возможная MITM-атака).

# 5. Аутентификация пользователя

## 5.1 По паролю
Самый простой способ. Пароль передаётся уже внутри зашифрованного канала (после шага 6 handshake), так что перехватить его нельзя. Но уязвим к брутфорсу.

```bash
ssh user@server.com        # спросит пароль
```
## 5.2 По ключу (Public Key Authentication) — рекомендуется
Основан на асимметричной криптографии. Приватный ключ **никогда не покидает** клиента.
**Как работает:**
```
Клиент                                    Сервер
  │                                           │
  │──── "хочу войти как user, вот мой pubkey"►│
  │                                           │  Сервер проверяет pubkey
  │                                           │  в ~/.ssh/authorized_keys
  │◄─── случайный challenge (зашифрован       │
  │      публичным ключом клиента)            │
  │                                           │
  │  Клиент расшифровывает приватным ключом   │
  │──── ответ на challenge ─────────────────► │
  │                                           │  Сервер проверяет ответ
  │◄─── SSH_MSG_USERAUTH_SUCCESS ───────────  │  ✓ Аутентификация пройдена
```
**Генерация ключей:**
```bash
ssh-keygen -t ed25519 -C "комментарий"     # создать пару ключей Ed25519
ssh-keygen -t rsa -b 4096                  # RSA 4096 (если нужна совместимость)
```
Результат:
```
~/.ssh/id_ed25519       ← приватный ключ (никому не отдавать!)
~/.ssh/id_ed25519.pub   ← публичный ключ (можно раздавать)
```
**Установка ключа на сервер:**
```bash
ssh-copy-id user@server.com
# или вручную:
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys   # на сервере
```
**Файлы:**

| Файл | Где | Назначение |
|------|-----|-----------|
| `~/.ssh/id_ed25519` | Клиент | Приватный ключ |
| `~/.ssh/id_ed25519.pub` | Клиент | Публичный ключ |
| `~/.ssh/authorized_keys` | Сервер | Разрешённые публичные ключи |
| `~/.ssh/known_hosts` | Клиент | Известные host_key серверов |
| `~/.ssh/config` | Клиент | Настройки подключений |
## 5.3 SSH-агент
Агент хранит расшифрованные приватные ключи в памяти — не нужно вводить passphrase при каждом подключении.
```bash
eval $(ssh-agent)          # запустить агент
ssh-add ~/.ssh/id_ed25519  # добавить ключ в агент
ssh-add -l                 # список загруженных ключей
```
**Agent Forwarding** (`-A`) — пробрасывает агент на удалённый сервер, чтобы с него можно было подключаться дальше, не копируя приватный ключ:
```bash
ssh -A user@bastion         # с bastion можно ssh дальше, используя твои ключи
```
> [!WARNING] Осторожно с Agent Forwarding
> Если на промежуточном сервере есть root или злоумышленник — он может воспользоваться проброшенным агентом. Включай `-A` только на доверенных хостах.
## 5.4 Другие методы

| Метод | Описание |
|-------|----------|
| `keyboard-interactive` | Многошаговая аутентификация (OTP, 2FA) |
| `GSSAPI` | Интеграция с Kerberos/Active Directory |
| `certificate` | SSH-сертификаты — ключи, подписанные CA (для крупных инфраструктур) |

# 6. SSH-конфиг (~/.ssh/config)
Позволяет задать параметры подключения для каждого хоста — не нужно каждый раз писать длинные команды.
```
Host bastion
    HostName 1.2.3.4
    User ubuntu
    Port 2222
    IdentityFile ~/.ssh/id_ed25519

Host prod-server
    HostName 10.0.0.5
    User deploy
    ProxyJump bastion          # подключиться через bastion
    IdentityFile ~/.ssh/id_rsa

Host *
    ServerAliveInterval 60     # keep-alive каждые 60 сек
    ServerAliveCountMax 3
```
```bash
ssh bastion          # вместо: ssh -p 2222 -i ~/.ssh/id_ed25519 ubuntu@1.2.3.4
ssh prod-server      # автоматически прыгнет через bastion
```

# 7. Туннели и Port Forwarding
> SSH умеет **оборачивать** любой TCP-трафик в зашифрованный канал. Это называется туннелированием.
## 7.1 Local Port Forwarding (-L)
Проброс порта с **локальной машины** на удалённый хост через SSH-сервер.
```
ssh -L [local_addr:]local_port:target_host:target_port user@ssh_server
```
**Пример:** база данных на удалённом сервере недоступна снаружи, только с самого сервера:
```bash
ssh -L 5432:localhost:5432 user@server.com
# теперь подключение к localhost:5432 → идёт через SSH → на server.com:5432
```
```
Твой ПК                   SSH-сервер              БД
localhost:5432 ──SSH──► server.com ──local──► localhost:5432
```
## 7.2 Remote Port Forwarding (-R)
Проброс порта с **удалённого** сервера на локальную машину. Позволяет открыть доступ к локальному сервису извне.
```
ssh -R [remote_addr:]remote_port:target_host:target_port user@ssh_server
```
**Пример:** показать локальный веб-сервер коллеге через публичный сервер:
```bash
ssh -R 8080:localhost:3000 user@server.com
# теперь server.com:8080 → через SSH → твой localhost:3000
```
```
Коллега ──► server.com:8080 ──SSH──► Твой ПК:3000
```
## 7.3 Dynamic Port Forwarding / SOCKS-прокси (-D)
Создаёт локальный **SOCKS5-прокси**. Весь трафик, направленный через него, уходит через SSH-сервер. Аналог VPN.
```bash
ssh -D 1080 user@server.com
# настрой браузер/приложение на SOCKS5 proxy localhost:1080
# весь их трафик пойдёт через server.com
```
```
Твой браузер ──SOCKS5──► localhost:1080 ──SSH──► server.com ──► интернет
```
## 7.4 ProxyJump / Jump Host (-J)
Подключение к серверу через промежуточный хост (бастион). SSH сам строит цепочку туннелей.
```bash
ssh -J bastion.com user@internal-server
# или несколько прыжков:
ssh -J bastion1.com,bastion2.com user@target
```
## 7.5. Сравнение туннелей

| Тип | Флаг | Направление | Типичный сценарий |
|-----|------|-------------|------------------|
| Local | `-L` | Локальный порт → удалённый | Доступ к БД/сервису за фаерволом |
| Remote | `-R` | Удалённый порт → локальный | Показать локальный сервис снаружи |
| Dynamic | `-D` | SOCKS-прокси | Обход блокировок, замена VPN |
| ProxyJump | `-J` | Цепочка SSH-хостов | Бастионы, многоуровневые сети |

# 8. Передача файлов
SSH лежит в основе нескольких протоколов передачи файлов:
```bash
# SCP — простое копирование (устаревает)
scp file.txt user@server:/path/
scp -r dir/ user@server:/path/

# SFTP — интерактивная сессия (рекомендуется)
sftp user@server
> put file.txt
> get remote_file.txt
> ls, cd, mkdir ...

# rsync через SSH — синхронизация с дельта-передачей
rsync -avz -e ssh ./dir/ user@server:/path/
```

# 9. Безопасность SSH-сервера
Типичные настройки `/etc/ssh/sshd_config` для hardening:
```
Port 2222                        # сменить стандартный порт (защита от ботов)
PermitRootLogin no               # запретить вход под root
PasswordAuthentication no        # только ключи, никаких паролей
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
MaxAuthTries 3                   # максимум попыток аутентификации
LoginGraceTime 20                # 20 сек на вход
AllowUsers deploy ubuntu         # только конкретные пользователи
X11Forwarding no                 # отключить проброс GUI (если не нужен)
```

# 10. Полная картина — SSH поверх TCP

```
Приложение (терминал, rsync, git)
          │
          ▼
┌─────────────────────────────┐
│     SSH Connection          │  каналы, туннели
├─────────────────────────────┤
│     SSH Authentication      │  ключ / пароль
├─────────────────────────────┤
│     SSH Transport           │  AES-256, Ed25519, ECDH
└─────────────────────────────┘
          │  зашифрованный поток байт
          ▼
┌─────────────────────────────┐
│     TCP (порт 22)           │  надёжная доставка
└─────────────────────────────┘
          │
          ▼
        IP → Ethernet → физика
```
**Хронология одного SSH-подключения:**
```
1. TCP SYN / SYN-ACK / ACK          (TCP handshake)
2. Обмен версиями SSH
3. Согласование алгоритмов
4. ECDH обмен ключами
5. Проверка host_key сервера
6. Выработка сессионных ключей → весь трафик зашифрован
7. Аутентификация пользователя (ключ или пароль)
8. Открытие канала (shell / exec / sftp / tunnel)
9. Работа
10. SSH_MSG_DISCONNECT
11. TCP FIN / FIN-ACK               (TCP закрытие)
```
