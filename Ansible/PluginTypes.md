# 1. Control node
## 1.1. Lookup plugins
- **Где выполняется:** На управляющей машине (control node).
- **Зачем:** Получение данных из внешних источников до начала выполнения задач. Не меняют состояние.
- **Пример:** `file`, `env`, `url`, `password`.
- **Когда писать свой:** Нужно прочитать данные из базы данных, внешнего API или сложного файла перед запуском плейбука.
- **Синтаксис:** `{{ lookup('my_plugin', 'arg') }}`
## 1.2. Filter plugins
- **Где выполняется:** На управляющей машине (во время рендеринга шаблонов Jinja2).
- **Зачем:** Преобразование данных (строки, списки, словари).
- **Пример:** `to_json`, `upper`, `default`.
- **Когда писать свой:** Нужна сложная логика обработки переменных внутри шаблонов или плейбуков (как в вашем примере с `fill_string`).
- **Синтаксис:** `{{ variable | my_filter(arg) }}`
## 1.3. Test plugins
- **Где выполняется:** На управляющей машине.
- **Зачем:** Проверка условий. Возвращают `True` или `False`. Похожи на фильтры, но семантически предназначены для проверок.
- **Пример:** `is defined`, `is file`, `is regex`.
- **Когда писать свой:** Для сложных условий в блоках `when`.
- **Код:** Почти идентичен Filter-плагинам, но функция должна возвращать булево значение.
- **Синтаксис:**
```yaml
when: my_var | my_test('pattern') 
```
## 1.4. Callback plugins
- **Где выполняется:** На управляющей машине.
- **Зачем:** Контроль вывода (логов) Ansible. Перехват событий (старт задачи, успех, ошибка).
- **Пример:** `profile_tasks` (показывает время выполнения задач), `json` (вывод в JSON для CI/CD).
- **Когда писать свой:** Нужно отправлять логи в Slack/Telegram, сохранять результаты в базу данных или кастомизировать вывод в консоль.
- **Настройка:** Включаются в `ansible.cfg`: `stdout_callback = my_callback`.
## 1.5. Inventory plugins
- **Где выполняется:** На управляющей машине (перед запуском).
- **Зачем:** Динамическое получение списка хостов.
- **Пример:** `aws_ec2`, `azure_rm`, `docker`.
- **Когда писать свой:** Ваши серверы не имеют статических IP и их список нужно брать из внутренней CMDB или API облака, которое не поддерживается стандартно.
- **Формат:** Должны возвращать JSON структуру хостов и групп.
## 1.6. Vars plugin
- **Где выполняется:** На управляющей машине.
- **Зачем:** Автоматическая загрузка переменных для хостов/групп из внешних источников.
- **Пример:** `host_group_vars` (стандартный, читает файлы `host_vars/`).
- **Когда писать свой:** Нужно подтягивать переменные для хоста напрямую из Vault или Базы Данных без использования `lookup` в каждой задаче.
## 1.7. Action plugins
- **Где выполняется:** На управляющей машине (обертка над модулем).
- **Зачем:** Управление логикой выполнения модуля. Могут изменять аргументы перед отправкой на хост или обрабатывать результат.
- **Пример:** `debug`, `set_fact`, `async`.
- **Когда писать свой:** Очень редко. Нужно сложное поведение, которое нельзя реализовать обычным модулем (например, специфическая логика повторных попыток или обработка секретов перед отправкой).
## 1.8. Connection plugins
- **Где выполняется:** На управляющей машине (инициирует соединение).
- **Зачем:** Определяет, _как_ Ansible подключается к хосту.
- **Пример:** `ssh`, `local`, `docker`, `kubectl`, `winrm`.
- **Когда писать свой:** Нужно управлять устройствами, к которым нет SSH доступа (например, через специфический API или последовательный порт).
## 1.9. Cache plugins
- **Где выполняется:** На управляющей машине.
- **Зачем:** Кэширование фактов (`gather_facts`).
- **Пример:** `memory`, `jsonfile`, `redis`.
- **Когда писать свой:** Нужно хранить факты о серверах в специфическом хранилище для ускорения работы в больших инфраструктурах.

# 2. Target host
## 2.1. Modules
Задачи (`tasks`) в плейбуках вызывают модули.
- **Где выполняется:** На целевом хосте (target host). Код копируется по SSH, выполняется, результат возвращается.
- **Зачем:** Изменение состояния системы (установка пакетов, создание файлов, перезапуск сервисов).
- **Пример:** `apt`, `copy`, `systemd`, `docker_container`.
- **Когда писать свой:** Когда встроенных модулей не хватает для управления специфическим железом или API, и нужно именно изменять состояние на сервере.
- **Синтаксис:**
```yaml
tasks:
  - name: Мой модуль
    my_custom_module:
	  param: value
```

# 3.Итог
## 3.1. Итоговая таблица

| Тип плагина    | Где работает       | Возвращает             | Основной синтаксис       | Частота написания |
| -------------- | ------------------ | ---------------------- | ------------------------ | ----------------- |
| **Module**     | Целевой хост       | Изменение состояния    | `module: arg=val`        | Средняя           |
| **Lookup**     | Управляющая машина | Данные (список)        | `{{ lookup('x', 'y') }}` | Высокая           |
| **Filter**     | Управляющая машина | Преобразованные данные | `{{ x \| filter(y) }}`   | Высокая           |
| **Test**       | Управляющая машина | True / False           | `{{ x \| test(y) }}`     | Средняя           |
| **Callback**   | Управляющая машина | Ничего (логи)          | (в ansible.cfg)          | Низкая            |
| **Inventory**  | Управляющая машина | Список хостов (JSON)   | (в инвентаре)            | Средняя           |
| **Action**     | Управляющая машина | Результат модуля       | (внутренняя логика)      | Редко             |
| **Connection** | Управляющая машина | Соединение             | (в ansible.cfg)          | Редко             |
## 3.2.Расположение
```text
project/
├── ansible.cfg
├── playbook.yml
└── plugins/
    ├── lookup/
    │   └── my_lookup.py
    ├── filter/
    │   └── my_filter.py
    ├── test/
    │   └── my_test.py
    ├── modules/
    │   └── my_module.py
    └── callback/
        └── my_callback.py
```
В `ansible.cfg` нужно указать пути:
```ini
[defaults]
lookup_plugins = ./plugins/lookup
filter_plugins = ./plugins/filter
test_plugins = ./plugins/test
library = ./plugins/modules
callback_plugins = ./plugins/callback
```
