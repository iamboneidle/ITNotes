Модули и другие ресурсы Kubernetes обычно создаются путем публикации манифеста JSON или YAML в конечной точке API REST Kubernetes. Кроме того,
можно использовать другие, более простые способы создания ресурсов, такие как команда `kubectl run`, но они обычно позволяют настраивать только ограниченный набор свойств, далеко не все. Кроме того, определение всех объектов Kubernetes из файлов YAML позволяет хранить их в системе управления версиями со всеми преимуществами, которые она приносит.

## 1.1. Главные части определения модуля
Определение модуля состоит из нескольких частей. Во-первых, это версия API Kubernetes, используемая в YAML, и тип ресурса, который описывает YAML. Затем почти во всех ресурсах Kubernetes находятся три важные секции:
- _метаданные_ (metadata) – включают имя, пространство имен, метки и другую информацию о модуле;
- _спецификация_ (spec) – содержит фактическое описание содержимого модуля, например контейнеры модуля, тома и другие данные;
- _статус_ (status) – содержит текущую информацию о работающем модуле, такую как условие, в котором находится модуль, описание и статус каждого контейнера, внутренний IP модуля, и другую базовую информацию.

## 2. Создание простого YAML дескриптора для нового модуля
```yaml
apiVersion: v1 # <- версия KubeAPI
kind: Pod # <- параметр, который говорит, что описывается Pod
metadata: 
  name: kubia-manual # <- имя модуля
  spec:
    containers:
      – image: luksa/kubia # <- образ, на основе которого создать контейнер
        name: kubia # <- имя контейнера
        ports:
          – containerPort: 8080 # <- порт
            protocol: TCP
```
## 2.1. Указание портов контейнера
Указание портов в определении модуля является чисто информационным. Их пропуск не влияет на возможность подключения клиентов к модулю через порт. Если контейнер принимает подключения через порт, привязанный к адресу 0.0.0.0, другие модули всегда смогут подключаться к нему, даже если порт не указан в спецификации модуля явно. Но имеет смысл явно определять порты, чтобы каждый, кто использует ваш кластер, мог быстро увидеть, к каким портам обеспечивается доступ каждым модулем. Явное определение портов также позволяет назначить каждому порту имя, что может пригодиться.
## 2.2. Использование команды `kubectl create`
```shell
kubectl create -f kubia-manual.yaml
```
Команда `kubectl create -f` применяется для создания любого ресурса (не только модулей) из файла YAML или JSON.

# 3. Просмотр журналов приложения
Обычное приложение выдает логи в стандартный поток вывода. Находясь внутри контейнера, оно пишет логи внутри него. Посмотреть их можно с помощью команды `docker logs <container_name>`, подключившись на надо по ssh.
Однако Kubernetes предоставляет более удобный способ. На локальной машине можно просто выполнить команду `kubectls logs <module_name>`. Такая команды подходит, если в модуле находится только один контейнер. В общем случае лучше пользоваться такой командой:
```shell
kubectl logs <module_name> -c <container_name>
```
При удалении модуля его журналы также удаляются. Для того чтобы сделать журналы модуля доступными даже после удаления модуля, необходимо настроить централизованное общекластерное ведение журнала, в котором все журналы хранятся в центральном хранилище.

# 4. Отправка запросов в модуль
Kubernetes позволяет настроить переадресацию портов на модуль. Это делается с помощью команды `kubectl port-forward`. Следующая ниже команда перенаправит локальный порт 8888 вашей машины на порт 8080 модуля kubia-manual:
```shell
kubectl port-forward <module_name> 8888:8080
```
В другом терминале теперь можно использовать `curl` для отправки HTTP-запроса в модуль через прокси-сервер переадресации портов kubectl port-forward, работающий на localhost:8888:
```shell
curl localhost: 8888
```
Использование такой переадресации портов является эффективным способом тестирования отдельного модуля.
