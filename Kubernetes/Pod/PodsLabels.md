На этапе, когда в кластере развернуто очень много модулей требуется механизм, который помогает разбивать модули кластера на более мелкие подмножества. Таким механизмом является метка (label).
# 1. Знакомство с метками
Метка – это произвольная пара «ключ-значение», присоединяемая к ресурсу, которая затем используется при отборе ресурсов с по­мощью селекторов меток (ресурсы фильтруются на основе того, включают они метку, указанную в селекторе, или нет). Ресурс может иметь несколько меток, если ключи этих меток уникальны в пределах данного ресурса. Метки обычно прикрепляются к ресурсам при их создании, но можно также добавлять дополнительные метки или даже изменять значения существующих меток позже без необходимости повторного создания ресурса.
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: kubia-manual-v2
  labels:
    creation_method: manual # |<- метки
    env: prod               # |__
spec:
  containers:
    – image: luksa/kubia
      name: kubia
      ports:
        – containerPort: 8080
          protocol: TCP
```
Создаем ресурс:
```shell
kubectl create -f kubia-manual-with-labels.yaml
```
Команда `kubectl get pods` по умолчанию не выводит никаких меток, но их
можно увидеть с помощью параметра `--show-labels`.
Если интересуют только определенные метки, то вместо вывода всех
меток вы можете указать их с помощью переключателя -L и отобразить каждую из них в своем собственном столбце.

# 2. Изменение меток существующих модулей
Метки можно также добавлять и модифицировать на существующих модулях.
```shell
kubectl label po kubia-manual creation_method=manual
```

> [!info] 
> При изменении существующих меток нужно использовать параметр
> `--overwrite`. 

```shell
kubectl label po kubia-manual-v2 env=debug --overwrite
```

# 3. Перечисление подмножеств модулей посредством селекторов меток
Само прикрепление меток к ресурсам, чтобы вы могли видеть метки рядом с каждым ресурсом при выводе их списка, не так интересно. Но метки идут рука об руку с _селекторами меток_. Селекторы меток позволяют выбрать подмножество модулей, помеченных определенными метками, и выполнять операцию на этих модулях. Селектор меток – это критерий, который фильтрует ресурсы на основе того, содержат они определенную метку с определенным значением или нет.
Селектор меток может выбирать ресурсы в зависимости от того:
- содержит ли (или не содержит) ресурс метку с определенным ключом;
- содержит ли ресурс метку с определенным ключом и значением;
- содержит ли ресурс метку с определенным ключом, но со значением, не равным указанному вами.

Для того чтобы просмотреть все модули `creation_method=manual`, выполните следующие ниже действия:
```shell
kubectl get po -l creation_method=manual
```
Для того чтобы вывести список всех модулей, которые включают метку `env`, каким бы ни было его значение:
```shell
kubectl get po -l env
```
И те, которые не имеют метки `env`:
```shell
kubectl get po -l '!env'
```
Также можно использовать такие конструкции:
- `creation_method!=manual`, чтобы выбрать модули с меткой `creation_method` с любым значением, кроме `manual`;
- `env in (prod,devel)`, чтобы выбрать модули с меткой `env`, установленной в `prod` или `development`;
- `env notin (prod,devel)`, чтобы выбрать модули с меткой `env`, установленной в любое значение, кроме `prod` или `devel`.
Селектор может также включать несколько критериев, разделенных запятыми. Для того чтобы соответствовать селектору, ресурсы должны соответствовать всем меткам: `app=pc, rel=beta`.

# 4. Использование меток и селекторов для ограничения планирования модулей
Вообще говоря, одна из основных идей Kubernetes заключается как раз таки в отвязке приложений от инфраструктуры. Однако, конечно, существуют ситуации, когда требуется указать явно узел. Такое может случаться, когда инфраструктура неоднородна и те или иные узлы могут иметь разные характеристики в плане железа. Например, может требоваться SSD, а не обычный диск или более мощное GPU для отдельных приложений.
Однако даже в таких условиях лучше всего просто обозначить характеристики узла для Kubernetes, чтобы не привязать приложение жестко к определенному узлу.

Метки могут быть присоединены к любому объекту Kubernetes, включая узлы. Обычно, когда системные администраторы добавляют новый узел в кластер, они классифицируют узел, прикрепляя метки, определяющие тип оборудования, которое предоставляет узел, или что-то еще, что может пригодиться при планировании модулей.

Команда крепит метку `gpu=true` к узлу `gke-kubia-85f6-node-0rrx`:
```shell
kubectl label node gke-kubia-85f6-node-0rrx gpu=true
```
Теперь селектором можно найти узел по метке:
```shell
kubectl get nodes -l gpu=true
```
## 4.1. Приписывание модулей к определенным узлам
Для того чтобы попросить планировщик выбирать только те узлы, которые предоставляют GPU, добавьте селектор узлов в YAML модуля.
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: kubia-gpu
spec:
  nodeSelector:
    gpu: "true" # <- nodeSelector говорит разворачиваться на узлах с меткой gpu=true
  containers:
    – image: luksa/kubia
      name: kubia
```
## 4.2. Планирование размещения на один конкретный узел
Точно так же можно приписать модуль конкретному узелу, потому что каждый узел тоже имеет уникальную метку с ключом kubernetes.io/hostname и значение, установленное в фактическое хост-имя узла. Вместе с тем, в случае если узел отключен, установка поля nodeSelector в конкретный узел по метке хост-имени может привести к тому, что модуль будет непригодным для приписывания (планирования).
