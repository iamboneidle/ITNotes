# 1. Создание контроллера репликации
Создать можно YAML-файлом:
```yaml
apiVersion: v1
kind: ReplicationController #<- этот манифест определяет контроллер репликации
metadata:
  name: kubia #<- имя контроллерарепликации
spec:
  replicas: 3 #<- количество реплик
  selector:
    app: kubia #<- селектор модулей. определяет, скакими модулями работает
  template:                       # |<- Шаблон модуля для создания новых модулей
    metadata:                     # |
      labels:                     # |
        app: kubia                # |
    spec:                         # |
      containers:                 # |
        – name: kubia             # |
          image: luksa/kubia      # |
          ports:                  # |
            – containerPort: 8080 # |__
```
При отправке файла на сервер API Kubernetes создает новый контроллер репликации с именем kubia, который гарантирует, что три экземпляра модуля
всегда совпадают с селектором меток app=kubia.

Далее командой `kubectl`:
```shell
kubectl create -f kubia-rc.yaml
```
___
# 2. Удаление модулей из под контроллера репликации
```shell
kubectl delete pod kubia-53thy
```
При таком действии контроллер репликации автоматически пересоздаст удаленный под.
___
# 3. Получение информации о контроллере репликации
Этой командой получим список контроллеров репликации:
```shell
kubectl get rc
```
Такой командой увидим информацию о контроллере репликации:
```shell
kubectl describe rc kubia
```
___
# 4. Что именно заставляет контроллер создать новый модуль
В то время как контроллер репликации немедленно уведомляется об удаляемом модуле (сервер API допускает, чтобы клиенты наблюдали за изменениями в ресурсах и списках ресурсов), это не то, что заставляет его создавать
сменный модуль. Уведомление заставляет контроллер **проверить** фактическое
количество модулей и предпринять соответствующее действие.
___
# 5. Отклик на аварийный сбой узла
1. Kubernetes помечает модуль статусом `NotReady`;
2. Прежде чем перепланировать модуль k8s ожидает еще некоторое время на случай, если причина сбоя является просто проблемой в сети, статус `Unknown`;
3. Если модуль вернется, то статус снова станет `Ready`, если нет, то модуль будет перепланирован.
