# 1. VLAN
VLAN — это механизм **логического разделения одной физической сети на несколько изолированных сегментов** на уровне L2 (канальный уровень). Придуман чтобы не тянуть отдельный кабель для каждого департамента.
Базируется на стандарте **IEEE 802.1Q**. В Ethernet-фрейм вставляется **4-байтный тег** между полями Source MAC и EtherType:
```
| Destination MAC | Source MAC | 802.1Q Tag | EtherType | Payload | FCS |
                               |____________|
                               TPID (2б) + TCI (2б)
                               TCI = PCP(3б) + DEI(1б) + VID(12б)
```
**Порты на свиче:**
- **Access port** — принимает нетегированный трафик, добавляет тег при входе, снимает при выходе. Конечное устройство не знает о VLAN.
- **Trunk port** — пропускает тегированный трафик нескольких VLAN. Используется между свичами и роутерами.
- **Native VLAN** — единственный VLAN на транке, трафик которого идёт без тега (по умолчанию VLAN 1, source of evil).
## 1.1. Изоляция
Свич хранит **отдельную MAC-таблицу на каждый VLAN**. BUM-трафик (Broadcast, Unknown unicast, Multicast) флудится только внутри своего VLAN. Трафик между VLAN — только через L3 (роутер или SVI на L3-свиче).
**Плюсы:**
- Простота — работает на любом управляемом свиче
- Низкие накладные расходы (4 байта оверхед)
- Аппаратная коммутация, нет задержек
- Безопасность: изоляция без физического разделения
**Минусы**
- **Лимит 4094 VLAN** — критично для датацентров с тысячами тенантов
- **Ограничен одним L2-доменом** — не умеет нативно ходить через L3-сети (нужны костыли типа Q-in-Q / 802.1ad)
- **STP (Spanning Tree)** — для L2 нужен STP чтобы не было петель, а STP медленный и блокирует линки
- Масштабирование MAC-таблиц — большой broadcast domain = большая таблица и шторм

# 2. BGP
BGP (Border Gateway Protocol) — это **протокол маршрутизации между автономными системами (AS)**. Это буквально протокол, на котором держится весь интернет. Он не ищет кратчайший путь — он ищет **путь по политике**.
## 2.1. Автономная система (AS)
AS — это набор IP-префиксов под управлением одной организации с единой политикой маршрутизации. Каждая AS имеет **ASN (AS Number)**: 16-битный (1–65535) или 32-битный (RFC 6793). Диапазон 64512–65534 — приватные ASN.

Каждая страна — это **AS (Автономная система)**. Внутри страны — свои дороги, своя логика, своё управление. Снаружи страна известна только своими **пограничными точками и тем, какие сети за ней стоят**.
Конкретно:
- **Ростелеком** — одна AS
- **Google** — другая AS
- **Твой датацентр** — ещё одна AS
- **Твой интернет-провайдер** — ещё одна

Каждой AS присваивается **ASN (номер)**. Например, Google — AS15169, Ростелеком — AS12389.
**AS анонсирует:**
AS говорит соседям: _"за мной стоят вот эти IP-префиксы"_.
```
AS15169 (Google) говорит:
  "8.8.8.0/24 — это я"
  "142.250.0.0/15 — это тоже я"

AS12389 (Ростелеком) говорит:
  "212.188.0.0/16 — это я"
```
BGP это объявление **распространяет по всем AS**. В итоге каждый роутер в интернете знает — чтобы попасть на 8.8.8.8, надо идти в сторону AS15169.
### 2.1.1 Общение AS м/у собой
Между AS поднимается **eBGP сессия** (external BGP) — это TCP соединение между пограничными роутерами.
```
[Ростелеком AS12389] ←── eBGP сессия ──► [Google AS15169]
    пограничный роутер                    пограничный роутер
```
Они обмениваются: _"вот что есть у меня, вот что есть у тебя"_. И дальше каждый решает — принять этот маршрут или нет, передать его дальше или нет. Это и есть **политика маршрутизации**.
### 2.1.2 Общение внутри AS
Внутри одной AS тоже нужно распространить информацию о внешних маршрутах. Для этого используется **iBGP** (internal BGP) — та же технология, но между роутерами одной AS.

Проблема iBGP — каждый роутер должен знать о каждом. При 100 роутерах это 4950 сессий. Поэтому придумали **Route Reflector** — один центральный роутер который получает маршруты от всех и раздаёт всем. Как почтовый сортировочный центр.
## 2.2. Как устроен
BGP работает поверх **TCP, порт 179**. Это не UDP с hello-пакетами, а полноценная TCP-сессия — надёжная доставка обновлений гарантирована.
**Типы BGP:**
- **eBGP** (external) — между разными AS. Обычно прямое соединение, TTL=1 по умолчанию
- **iBGP** (internal) — внутри одной AS. TTL=255, нужен full mesh или route reflector

**BGP-сообщения:**

| Тип           | Назначение                                      |
| ------------- | ----------------------------------------------- |
| OPEN          | Установка сессии, обмен ASN, Hold Timer         |
| UPDATE        | Анонс новых префиксов или отзыв старых          |
| KEEPALIVE     | Поддержание сессии (по умолчанию каждые 60 сек) |
| NOTIFICATION  | Ошибка → сессия рвётся                          |
| ROUTE-REFRESH | Запрос переотправки RIB без сброса сессии       |
## 2.3. BGP-атрибуты
BGP выбирает маршрут **не по метрике**, а по набору атрибутов. Порядок выбора (упрощённо):
```
1. Weight (Cisco proprietary, локально)
2. Local Preference (внутри AS, выше = лучше)
3. Locally originated (network/redistribute)
4. AS Path length (короче = лучше)
5. Origin (IGP > EGP > Incomplete)
6. MED (Multi-Exit Discriminator, ниже = лучше)
7. eBGP > iBGP
8. IGP metric до next-hop
9. Router ID (ниже = лучше, тiebreaker)
```
Управляя этими атрибутами, операторы строят **traffic engineering** — куда трафик входит, куда выходит.
## 2.4. Конвергенция
BGP **медленный намеренно**. Hold Timer = 90 сек по умолчанию. После падения сессии — несколько минут до конвергенции. Это сделано чтобы не флудить интернет обновлениями при флапах. Есть BFD (Bidirectional Forwarding Detection) для быстрого детекта отказов на уровне секунд.
>Конвергенция — это момент когда **все роутеры в сети согласны с тем, как выглядит мир**.
>Представь: упал линк между двумя AS. Роутеры не знают об этом мгновенно. Какое-то время одни думают что путь есть, другие уже знают что нет. Сеть в **несогласованном состоянии** — это и есть отсутствие конвергенции.
>Конвергенция завершилась — все роутеры обновили таблицы и снова **единогласны**.

```
1. Линк упал между AS-A и AS-B
        │
        ▼
2. BGP Hold Timer истекает (90 секунд по умолчанию)
   "сосед не отвечал 90 сек — считаю его мёртвым"
        │
        ▼
3. AS-A отзывает маршруты которые шли через AS-B
   шлёт UPDATE соседям: "префикс X.X.X.X — withdraw"
        │
        ▼
4. Соседи получают withdraw, обновляют таблицы,
   шлют дальше своим соседям
        │
        ▼
5. Волна обновлений расходится по всему интернету
        │
        ▼
6. Все роутеры обновились — конвергенция завершена
```
## 2.5. Достоинства и недостатки
**Плюсы**
- Масштабируется до размеров всего интернета
- Гибкая политика маршрутизации
- Поддержка множества address families (IPv4, IPv6, VPN, EVPN...)
- Стабильность и предсказуемость
**Минусы**
- Медленная конвергенция
- Сложность конфигурации и troubleshooting
- iBGP требует full mesh или RR (route reflector) — дополнительная сложность
- Нет встроенной защиты от route hijacking (частично решается RPKI)

# 3. VXLAN
VXLAN (Virtual Extensible LAN (Local Area Network)) решает главную проблему VLAN — **масштабирование L2 поверх L3**. Это туннельный протокол: берёт целый Ethernet-фрейм и упаковывает его в UDP. L2 поверх L3. Overlay поверх Underlay.
## 3.1. Инкапсуляция
```
[ Outer Ethernet | Outer IP | Outer UDP | VXLAN Header | Inner Ethernet Frame ]
                              dst:4789   |____________|
                                         Flags + VNI (24 бит)
```
**VNI (VXLAN Network Identifier)** — 24 бита → **16 миллионов сегментов** (против 4094 у VLAN).
Outer IP — это адреса **VTEP (VXLAN Tunnel Endpoint)** — виртуальных точек входа/выхода в туннель. VTEP может быть физическим свичом или виртуальным (например, Open vSwitch в гипервизоре).
## 3.2. VTEP и как работает пересылка
1. VM1 на хосте A отправляет фрейм VM2 на хосте B
2. VTEP хоста A смотрит: знает ли он, за каким VTEP сидит MAC VM2?
3. Если знает → инкапсулирует фрейм, отправляет UDP на IP VTEP хоста B
4. VTEP хоста B → декапсулирует → доставляет VM2 нативный Ethernet-фрейм
5. VM2 ничего не знает о VXLAN
### 3.2.1. Как VTEP узнаёт о других VTEP и MAC?
**1. Multicast (data plane learning)**
- Каждый VNI маппится на multicast-группу
- BUM-трафик флудится через multicast в underlay
- VTEP учится: "вот этот MAC пришёл от вот этого VTEP"
- Минус: нужна поддержка multicast в underlay, сложно
**2. BGP EVPN (control plane learning)** ← современный стандарт
- VTEP анонсирует свои MAC/IP через BGP
- Используется address family **L2VPN EVPN (RFC 7432)**
- BUM-трафик — через ingress replication (юникаст на каждый VTEP)
- Минус: сложнее конфиг, нужен BGP везде
### 3.2.2. BGP EVPN — связка VXLAN+BGP
Это позволяет строить **distributed anycast gateway** — одинаковый виртуальный шлюз на каждом leaf-свиче, без centralised routing.
**Оверхед VXLAN:**
```
Outer Ethernet:  14 байт
Outer IP:        20 байт
Outer UDP:        8 байт
VXLAN Header:     8 байт
─────────────────────────
Итого оверхед:   50 байт
```
При стандартном MTU 1500 байт → inner MTU = 1450. Нужно либо увеличивать MTU в underlay до **1600+ байт (Jumbo Frames)**, либо страдать от фрагментации.
**Плюсы**
- 16 млн VNI — мультиарендность датацентра без ограничений
- L2 поверх L3 — VM мигрирует между хостами, сохраняя IP и MAC
- Работает поверх любой IP-сети (не нужен L2 underlay)
- С BGP EVPN — масштабируемый control plane
**Минусы**
- *Оверхед 50 байт* — нужны Jumbo Frames в underlay
- Сложность: нужно понимать и overlay и underlay одновременно
- Troubleshooting сложнее — трафик скрыт внутри туннеля
- Требует VTEP-способного железа (или CPU в случае soft VTEP)
- BGP EVPN — дополнительный уровень сложности конфигурации

# 4. Сравнение
## 4.1. Сравнительная таблица

|                    | VLAN            | BGP              | VXLAN                |
| ------------------ | --------------- | ---------------- | -------------------- |
| **Уровень OSI**    | L2              | L3               | L2 over L3 (overlay) |
| **Масштаб**        | 4094 сегмента   | Весь интернет    | 16M сегментов        |
| **Транспорт**      | Ethernet        | TCP/179          | UDP/4789             |
| **Применение**     | LAN-сегментация | Inter-AS routing | DC overlay           |
| **Сложность**      | Низкая          | Высокая          | Высокая              |
| **Оверхед**        | 4 байта         | Нет              | 50 байт              |
| **Мобильность VM** | Нет             | Нет              | Да                   |
## 4.2. Как они работают вместе
```
┌─────────────────────────────────────────┐
│           OVERLAY (VXLAN)               │
│  VM ── VTEP ══════════════ VTEP ── VM   │
│         │    VNI туннели    │           │
├─────────┼───────────────────┼───────────┤
│           UNDERLAY (BGP)                │
│  Leaf ── Spine ── Spine ── Leaf         │
│    └── BGP Unnumbered везде ──┘         │
├─────────────────────────────────────────┤
│           ФИЗИКА                        │
│  VLAN используется внутри одного rack   │
│  для management / storage сетей         │
└─────────────────────────────────────────┘
```
- **VLAN** — сегрегация трафика на физическом уровне внутри стойки
- **BGP** — строит underlay IP-сеть между всеми свичами, анонсирует VTEP-адреса, реализует EVPN control plane
- **VXLAN** — overlay поверх этой IP-сети, даёт L2-связность между VM на разных хостах

Это архитектура **Spine-Leaf с BGP EVPN/VXLAN** — стандарт де-факто в современных датацентрах (используют Cisco, Arista, Juniper, Nokia, а также open-source стеки типа FRR + OVS).

# 5. Overlay & Underlay
**Overlay** — это когда **верхний уровень эмулирует нижний**, упаковывая его внутрь себя.
```
L2 фрейм (inner)   ← то что эмулируется
    ↓ упакован в
UDP пакет (L4)     ← то что реально везёт
    ↓ внутри
IP пакет (L3)
    ↓ внутри
Ethernet (L2)
```
L2 **едет внутри** L4. Это и есть overlay — ты берёшь протокол нижнего уровня и **туннелируешь его поверх** протокола верхнего уровня.
Физическая сеть видит только внешний UDP/IP и понятия не имеет что внутри ещё один Ethernet-фрейм.

## 5. 1. Терминология
- **Underlay** — реальная физическая сеть, которая реально маршрутизирует пакеты (IP/BGP)
- **Overlay** — виртуальная сеть поверх неё, которая эмулирует нужное поведение (L2 домен через VXLAN)
## 5.2. Другие примеры overlay-сетей
- **GRE** — IP туннель поверх IP
- **IPsec/VPN** — зашифрованный туннель поверх IP
- **MPLS** — метки поверх IP (спорно, но концептуально похоже)
- **WireGuard/OpenVPN** — L3 поверх UDP

# 6. Туннели
Представь две точки A и B. Между ними — чужая сеть, которая понимает только IP. А тебе нужно прогнать через неё Ethernet-фреймы (которые IP не являются).

Что делаешь:
```
A                    чужая сеть                    B
│                                                   │
│  [Ethernet фрейм]                                 │
│       ↓                                           │
│  заворачиваешь в IP пакет                         │
│  [IP | Ethernet фрейм]                            │
│       ↓                                           │
│  отправляешь через чужую сеть ────────────────►   │
│                                     разворачиваешь│
│                       получаешь оригинальный фрейм│
```
Чужая сеть видит обычный IP пакет. Она его просто везёт. Что внутри — не знает и не проверяет. Это и есть туннель — **труба внутри сети**.

Точки A и B — это и есть **VTEP** в случае [[#3. VXLAN|VXLAN]]. Они умеют паковать и распаковывать. Всё что между ними — тупо везёт пакеты не глядя внутрь.
