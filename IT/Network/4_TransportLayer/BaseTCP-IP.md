# 2. Установка TCP соединения
>**Three-way handshake** — процесс из трёх сегментов, после которого обе стороны знают порядковые номера друг друга и готовы к обмену данными.

```
Клиент                          Сервер
  |                               |
  |──── SYN (seq=x) ─────────────►|  [1] Запрос соединения
  |                               |      Сервер запоминает seq,
  |                               |      создаёт сокет (буферы + структуры)
  |                               |
  |◄─── SYN + ACK (seq=y, ack=x+1)|  [2] Подтверждение + свой seq
  |                               |      Сервер → SYN-RECEIVED
  |                               |
  |──── ACK (ack=y+1) ───────────►|  [3] Финальное подтверждение
  |                               |
ESTABLISHED ◄─────────────────► ESTABLISHED
```
## 2.1. Детали каждого шага
### 2.1.1. SYN (Клиент -> сервер)
- Клиент посылает сегмент с флагом `SYN` и своим начальным `seq=x`
- `SYN` устанавливается **только в первом пакете** каждой стороны — это флаг синхронизации
- Сервер получает сегмент, запоминает `seq` и пытается создать **сокет** (буферы + управляющие структуры памяти)
    - Успех → отвечает `SYN + ACK`, переходит в `SYN-RECEIVED`
    - Неудача → отвечает `RST` (сброс соединения)
### 2.1.2. SYN + ACK (сервер -> клиент)
- Сервер подтверждает (`ACK = x+1`) и сообщает свой начальный `seq=y`
- Сервер переходит в состояние `SYN-RECEIVED`
### 2.1.3. ACK (клиент -> сервер)
- Клиент получает `SYN + ACK`, запоминает `seq` сервера, отправляет `ACK = y+1`
- Клиент переходит в `ESTABLISHED`
- Сервер, получив `ACK` → переходит в `ESTABLISHED`

> [!info] 
> `seq=x` — это **порядковый номер** (sequence number) сегмента.
> Когда TCP открывает соединение, каждая сторона выбирает случайное начальное число — **ISN (Initial Sequence Number)**. В диаграмме это просто обозначено как `x` и `y`, потому что конкретное значение не важно для понимания процесса.
> **Зачем он нужен**: 
> Представь, что ты отправляешь файл в 3 сегментах. Сегменты могут прийти не по порядку или дублироваться. `seq` позволяет получателю собрать их в правильном порядке и понять, какие уже получены.
> ```
> Отправитель                    Получатель
  │                               │
  │──── сегмент, seq=1000 ───────►│  байты 1000–1499
  │──── сегмент, seq=1500 ───────►│  байты 1500–1999
  │──── сегмент, seq=2000 ───────►│  байты 2000–2499
  │                               │
  │◄─── ACK, ack=2500 ────────────│  "получил всё до 2500, давай дальше"
> ```
> **Почему ISN случайный, а не всегда 0?** Безопасность — чтобы злоумышленник не мог предсказать номер и подделать сегменты чужого соединения.
## 2.2. Состояния и исключения
| Ситуация | Поведение |
|----------|-----------|
| Клиент не получает ответ | Повторяет процесс через **10 секунд** |
| Сервер получает `RST` вместо `ACK` | Закрывает сокет → `CLOSED` |
| Сервер не получает `ACK` (тайм-аут) | Закрывает сокет → `CLOSED` |
| Клиент получает `RST` от сервера | Прекращает попытки соединения |

> [!note] Почему три, а не четыре сегмента?
> Теоретически возможен вариант из 4 сегментов: `SYN→`, `←ACK`, `←SYN`, `ACK→`. На практике сервер объединяет `ACK` и `SYN` в один пакет `SYN+ACK` — экономия времени.
## 2.3. Структура заголовка TCP
[[UDP#2. Структура заголовка UDP]]
Минимальный размер заголовка TCP составляет **20 байт**. Если используются дополнительные опции, размер может увеличиваться до **60 байт**.

|**Поле**|**Размер**|**Описание**|
|---|---|---|
|**Source Port**|16 бит|Порт отправителя.|
|**Destination Port**|16 бит|Порт получателя.|
|**Sequence Number**|32 бита|Порядковый номер первого байта данных в этом сегменте. Нужен для сборки пакетов в правильном порядке.|
|**Acknowledgment Number**|32 бита|Номер подтверждения. Указывает порядковый номер байта, который отправитель ожидает получить следующим.|
|**Data Offset**|4 бита|Длина заголовка (измеряется в 32-битных словах). Показывает, где заканчивается заголовок и начинаются данные.|
|**Reserved**|3 бита|Зарезервировано для будущего использования (всегда 0).|
|**Flags (Control Bits)**|9 бит|Управляющие флаги (см. ниже).|
|**Window Size**|16 бит|Размер приемного окна (rwnd). Количество байт, которое получатель готов принять.|
|**Checksum**|16 бит|Контрольная сумма для проверки целостности всего сегмента (заголовок + данные).|
|**Urgent Pointer**|16 бит|Указатель на «срочные» данные (используется редко, вместе с флагом URG).|
|**Options**|0-40 байт|Дополнительные параметры (например, MSS, Window Scaling, SACK).|

# 3. Поддержка соединения
> **Keep-Alive** — механизм проверки живости неактивного соединения. Периодически отправляет пустой зондирующий сегмент партнёру.

```
Сторона A              Сторона B
  |                        |
  |  (тишина... 2 часа)    |
  |                        |
  |──── keep-alive ───────►|
  |◄─── ACK ───────────────|  ✓ Партнёр жив
  |                        |
  |──── keep-alive ───────►|
  |   (нет ответа)         |  ✗ Партнёр недоступен → закрыть соединение
```

| Параметр                    | Значение по умолчанию | Описание                                             |
| --------------------------- | --------------------- | ---------------------------------------------------- |
| Тайм-аут до первой проверки | **2 часа**            | Время бездействия перед отправкой keep-alive         |
| Интервал между зондами      | ~75 секунд            | Как часто повторять, если нет ответа                 |
| Число попыток               | 9                     | После 9 неответов — соединение считается разорванным |
- Keep-alive **не является обязательным** — это опциональный механизм
- Приложение может **отключить** его самостоятельно через socket-опции (`SO_KEEPALIVE`)
- 2 часа по умолчанию — очень большое значение; в реальных приложениях (HTTP, SSH) обычно задают значительно меньше
- Сервер может не поддерживать keep-alive — тогда зонды просто игнорируются

# 4. Завершение соединения

> **Four-way handshake** — корректное закрытие соединения. Каждая сторона закрывает свой поток данных независимо (`FIN`), что позволяет завершить передачу без потерь.

```
Клиент                          Сервер
  |                               |
  |──── FIN (seq=u) ─────────────►|  [1] Клиент закончил отправку
  |                               |      Сервер → CLOSE-WAIT
  |◄─── ACK (ack=u+1) ─────────── |  [2] Сервер подтверждает
  |                               |      (сервер ещё может отправлять данные)
  |                               |
  |◄─── FIN (seq=v) ──────────────|  [3] Сервер тоже закончил
  |                               |      Клиент → TIME-WAIT
  |──── ACK (ack=v+1) ───────────►|  [4] Клиент подтверждает
  |                               |
  |  TIME-WAIT (2×MSL)            |
  |  затем CLOSED ◄──────────────► CLOSED
```

| Состояние | Кто | Описание |
|-----------|-----|----------|
| `FIN-WAIT-1` | Инициатор | Отправил `FIN`, ждёт `ACK` |
| `FIN-WAIT-2` | Инициатор | Получил `ACK`, ждёт `FIN` от другой стороны |
| `CLOSE-WAIT` | Получатель | Получил `FIN`, может ещё отправлять данные |
| `LAST-ACK` | Получатель | Отправил свой `FIN`, ждёт финального `ACK` |
| `TIME-WAIT` | Инициатор | Ждёт `2×MSL` перед окончательным закрытием |

> [!NOTE] Зачем нужен TIME-WAIT?
> `TIME-WAIT` длится **2×MSL** (Maximum Segment Lifetime, обычно 2 мин = итого 4 мин). Это защита от ситуации, когда запоздавший сегмент старого соединения попадёт в новое с теми же портами.

Вместо корректного `FIN` можно отправить `RST` — немедленный сброс без подтверждений:
- Используется при ошибке или отказе
- Получатель `RST` должен немедленно закрыть соединение
- Данные в буфере теряются

# 5. Краткая диаграмма состояний
```
         CLOSED
           │
     активное открытие (SYN sent)
           ▼
       SYN-SENT ──► (получен RST) ──► CLOSED
           │
    получен SYN+ACK
           ▼
       ESTABLISHED ◄──── (пассивное открытие через SYN-RECEIVED)
           │
      активное закрытие (FIN sent)
           ▼
      FIN-WAIT-1 ──► FIN-WAIT-2 ──► TIME-WAIT ──► CLOSED
```

# 7. Механизмы внутри TCP
## 7.1. Window scaling
Стандартный заголовок TCP использует 16 бит для определения размера окна , что ограничивает его максимальный размер до **64 КБ**. В современных скоростных сетях этого катастрофически мало (простой простаивает в ожидании подтверждений).
- **Как работает:** Опция `Window Scale` используется во время 3-этапного рукопожатия (SYN). Она задает множитель (степень двойки), на который нужно умножить значение в 16-битном поле.
- **Результат:** Позволяет увеличить размер окна до **1 ГБ**.
- **Зачем это нужно:** Позволяет передавать больше данных без ожидания ACK, что критично для LFN (Long Fat Networks) — сетей с высокой пропускной способностью и большой задержкой.
## 7.2. TCP Fast Open
Классический TCP требует завершения "трехэтапного рукопожатия" (SYN -> SYN-ACK -> ACK) до того, как начнется передача полезных данных (HTTP-запроса и т.д.). Это добавляет лишнюю задержку (RTT).
- **Как работает:**
    1. При первом подключении клиент получает от сервера специальный **Cookie**.
    2. При повторном подключении клиент отправляет пакет `SYN` сразу вместе с `Cookie` и **первыми данными** (например, GET-запросом).
    3. Сервер подтверждает SYN и данные в одном пакете.
- **Результат:** Экономит ровно один цикл RTT при повторных соединениях.
- **Включение в Linux:**
    - Проверить: `cat /proc/sys/net/ipv4/tcp_fastopen`
    - Значение `3` разрешает и клиентский, и серверный TFO.
## 7.3. TCP Slow Start
Это часть алгоритма предотвращения перегрузки (Congestion Control). TCP не знает пропускную способность сети в начале сессии и не может сразу "влить" гигабит данных.
- **Как работает:**
    1. Вводится понятие **Congestion Window (cwnd)** — окно перегрузки на стороне отправителя.
    2. В начале `cwnd` обычно равен 10 сегментам (IW10).
    3. При получении каждого подтверждения (ACK) размер окна увеличивается **экспоненциально** (удваивается за каждый RTT).
    4. Рост продолжается до тех пор, пока не возникнет потеря пакета или не будет достигнут порог `ssthresh` (Slow Start Threshold), после чего рост становится линейным.
- **Зачем это нужно:** Предотвращает коллапс сети, если пропускная способность узкого канала меньше, чем скорость отправки данных хостом.
## 7.4. TCP Sliding Window
Без скользящего окна отправителю пришлось бы ждать подтверждения (ACK) для каждого отправленного пакета, прежде чем отправить следующий. Это крайне неэффективно.
**Sliding Window** позволяет отправителю передавать сразу несколько пакетов («окно»), не дожидаясь ACK для каждого из них.

Представь длинную ленту байтов данных. Окно — это «рамка», которая накладывается на часть этой ленты:
1. **Sent and Acknowledged**: Данные слева от окна (уже ушли и подтверждены).
2. **Sent but not Acknowledged**: Данные внутри окна, которые уже улетели в сеть, но подтверждение на них еще не пришло.
3. **Ready to Send**: Данные внутри окна, которые можно отправлять прямо сейчас.
4. **Not usable**: Данные справа от окна, которые нельзя отправлять, пока окно не сдвинется.
Когда приходит ACK для самого старого байта в окне, рамка **сдвигается (slide)** вправо, позволяя отправить следующие байты.

Механизм регулируется двумя параметрами:
- **RWND (Receive Window)**: Сообщается _получателем_. Он говорит: «У меня в буфере осталось всего X места, не присылай больше этого объема, иначе я просто отброшу пакеты». Это защищает получателя от переполнения.
- **CWND (Congestion Window)**: Определяется _отправителем_ на основе состояния сети (тот самый Slow Start). Он говорит: «Сеть сейчас перегружена, я буду отправлять по чуть-чуть, чтобы не создать затор».
