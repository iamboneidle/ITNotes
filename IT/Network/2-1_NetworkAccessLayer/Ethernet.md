Ethernet, VLAN, STP — канальный уровень
> **Ethernet (IEEE 802.3)** — семейство технологий канального уровня для проводных локальных сетей. Определяет формат кадра, адресацию (MAC), метод доступа к среде и скорости передачи.

___
# 1. MAC-адрес
**MAC (Media Access Control)** — физический адрес сетевого интерфейса. Назначается производителем и «зашит» в железо, хотя программно его можно подменить.
```
  FF : FF : FF : FF : FF : FF
  └──────────┘   └──────────┘
     OUI              NIC
  (производитель)  (номер устройства)
```
- Длина: **6 байт (48 бит)**, записывается в hex: `00:1A:2B:3C:4D:5E`
- **OUI** (Organizationally Unique Identifier) — первые 3 байта, выдаются IEEE производителю
- **NIC** — последние 3 байта, назначает сам производитель
- `FF:FF:FF:FF:FF:FF` — широковещательный адрес (broadcast), получают все в сегменте
- Адрес с установленным младшим битом первого байта — multicast

# 2. Структура Ethernet-фрейма (IEEE 802.3)

```
 ┌──────────┬─────────┬──────────┬──────────┬────────────────┬─────┐
 │ Преамбула│  SFD    │  DST MAC │  SRC MAC │  EtherType/Len │ FCS │
 │  7 байт  │ 1 байт  │  6 байт  │  6 байт  │    2 байта     │4 байт│
 └──────────┴─────────┴──────────┴──────────┴────────────────┴─────┘
                                              └──── Данные (46–1500 байт) ──┘
```

| Поле               | Размер       | Описание                                                                  |
| ------------------ | ------------ | ------------------------------------------------------------------------- |
| Преамбула          | 7 байт       | Чередование `10101010...` — синхронизация получателя                      |
| SFD                | 1 байт       | Start Frame Delimiter (`10101011`) — граница начала фрейма                |
| DST MAC            | 6 байт       | MAC-адрес получателя                                                      |
| SRC MAC            | 6 байт       | MAC-адрес отправителя                                                     |
| EtherType / Length | 2 байта      | ≥ `0x0800` → тип протокола (IPv4, IPv6, ARP); < `0x0600` → длина данных   |
| Данные (Payload)   | 46–1500 байт | Полезная нагрузка. Минимум 46 байт — если меньше, добавляется **padding** |
| FCS                | 4 байта      | Frame Check Sequence — контрольная сумма CRC-32, обнаружение ошибок       |

**MTU (Maximum Transmission Unit)** — максимальный размер полезной нагрузки: **1500 байт**.  
Полный размер фрейма с заголовками: 1518 байт (или 1522 с тегом VLAN).
### Распространённые значения EtherType

| Значение | Протокол |
|----------|----------|
| `0x0800` | IPv4 |
| `0x0806` | ARP |
| `0x86DD` | IPv6 |
| `0x8100` | VLAN (802.1Q) |
| `0x8847` | MPLS |

# 3. Метод доступа к среде — CSMA/CD

> **CSMA/CD** (Carrier Sense Multiple Access / Collision Detection) — метод контроля доступа к общей среде передачи. Актуален для **полудуплексных** соединений и старых топологий с хабами.
### Алгоритм
```
1. Слушай среду (Carrier Sense)
        │
        ▼
   Среда свободна? ──── Нет ───► Жди, повторяй прослушивание
        │
       Да
        │
        ▼
2. Начни передачу
        │
        ▼
3. Обнаружена коллизия? ──── Нет ───► Передача завершена ✓
        │
       Да
        │
        ▼
4. Отправь JAM-сигнал (32 бита) — оповести всех о коллизии
        │
        ▼
5. Выжди случайное время (Binary Exponential Backoff)
        │
        ▼
6. Повтори с шага 1 (максимум попыток, затем ошибка)
```
### Binary Exponential Backoff
После каждой коллизии узел ждёт случайное число слотов от `0` до `2^n - 1`, где `n` — номер попытки.

| Попытка | Диапазон ожидания |
| ------- | ----------------- |
| 1       | 0–1 слот          |
| 2       | 0–3 слота         |
| 3       | 0–7 слотов        |
| 10+     | 0–1023 слота      |

> [!NOTE] CSMA/CD в современных сетях
> В сетях с коммутаторами (full-duplex) коллизий не бывает — каждый порт изолирован. CSMA/CD фактически не используется. Актуален только при подключении через хаб (редкость) или в специфических условиях.

# 4. Сетевые устройства канального уровня
## 4.1. Хаб (Hub) — L1

- Повторяет сигнал на **все** порты без разбора
- Все устройства в одном **домене коллизий**
- Не понимает MAC-адресов
- Устарел, почти не используется
## 4.2. Коммутатор (Switch) — L2
- Пересылает фреймы **только** на нужный порт по MAC-адресу
- Каждый порт — отдельный **домен коллизий**
- Ведёт **таблицу MAC-адресов** (CAM-таблица)
- Поддерживает full-duplex → CSMA/CD не нужен
#### Как коммутатор учит MAC-адреса
```
1. Фрейм пришёл на порт 3 от MAC AA:BB:CC:...
   → Запись в таблицу: AA:BB:CC:... → порт 3

2. Куда отправить фрейм?
   → DST MAC есть в таблице → отправить на конкретный порт (unicast)
   → DST MAC НЕТ в таблице → отправить на ВСЕ порты, кроме входящего (flooding)
   → DST MAC = FF:FF:FF:FF:FF:FF → broadcast, отправить на все порты

3. Записи в таблице устаревают (~5 минут бездействия) и удаляются
```
#### Режимы коммутации

| Режим | Описание | Задержка | Обнаружение ошибок |
|-------|----------|----------|--------------------|
| **Store-and-Forward** | Получает весь фрейм, проверяет FCS, затем пересылает | Высокая | Полное |
| **Cut-Through** | Пересылает сразу после чтения DST MAC | Низкая | Нет |
| **Fragment-Free** | Ждёт первые 64 байта (отсеивает большинство коллизий) | Средняя | Частичное |

## 5. ARP — связь MAC и IP
> **ARP (Address Resolution Protocol)** — протокол разрешения IP-адреса в MAC-адрес. Работает на границе L2/L3, технически относится к L2.
```
Кто знает MAC для 192.168.1.1?   (broadcast: FF:FF:FF:FF:FF:FF)
─────────────────────────────────────────────────────────────────►
                                       Это я! Мой MAC: AA:BB:CC:DD:EE:FF
◄─────────────────────────────────────────────────────────────────
```
- Результат кэшируется в **ARP-таблице** (arp cache) на несколько минут
- **Gratuitous ARP** — узел объявляет свой IP+MAC без запроса (используется при смене IP или для обнаружения конфликтов)
- **ARP Spoofing** — атака: злоумышленник рассылает поддельные ARP-ответы, перехватывая трафик
___
## 6. VLAN — виртуальные локальные сети (IEEE 802.1Q)
> **VLAN (Virtual LAN)** — логическое разбиение одного физического коммутатора на несколько изолированных сетей. Устройства в разных VLAN не видят друг друга на L2.
## 6.1.  Зачем нужны VLAN

- **Безопасность** — изоляция трафика
- **Производительность** — уменьшение broadcast-доменов
- **Гибкость** — логическая сегментация без физического переключения кабелей
## 6.2. Тегирование 802.1Q
При прохождении через транк к фрейму добавляется **4-байтовый тег**:
```
 ┌──────────┬─────────┬──────────┬──────────┬──────────────────────┬──────────────────┬──────┐
 │ Преамбула│   SFD   │  DST MAC │  SRC MAC │   802.1Q тег (4 байт)│ EtherType / Data │  FCS │
 └──────────┴─────────┴──────────┴──────────┴──────────────────────┴──────────────────┴──────┘
```
**Структура 802.1Q тега:**
```
 ┌────────────────┬───┬───┬──────────────────┐
 │  TPID (2 байт) │PCP│DEI│   VID (12 бит)   │
 │    0x8100      │3 б│1 б│   VLAN ID 1–4094 │
 └────────────────┴───┴───┴──────────────────┘
```

| Поле | Описание |
|------|----------|
| TPID | `0x8100` — маркер того, что это 802.1Q фрейм |
| PCP | Priority Code Point — приоритет (0–7), используется в QoS |
| DEI | Drop Eligible Indicator — можно ли сбросить при перегрузке |
| VID | VLAN ID, 12 бит → 4094 возможных VLAN (0 и 4095 зарезервированы) |
## 6.3. Типы портов коммутатора

| Тип | Описание | Тегирование |
|-----|----------|-------------|
| **Access** | Подключает конечное устройство, принадлежит одному VLAN | Теги снимаются/добавляются коммутатором, устройство не знает о VLAN |
| **Trunk** | Соединяет коммутаторы или маршрутизатор, несёт несколько VLAN | Фреймы идут с тегами 802.1Q |
| **Hybrid** | Комбинация: одни VLAN тегированные, другие нет | Зависит от настройки |
## 6.4. Native VLAN

На транк-порту один VLAN может быть **native** — его фреймы идут **без тега**. По умолчанию — VLAN 1.

> [!WARNING] Безопасность
> VLAN 1 как native VLAN считается небезопасной практикой. Рекомендуется сменить native VLAN на неиспользуемый номер.
## 6.5. Маршрутизация между VLAN — Router-on-a-Stick
Устройства в разных VLAN не могут общаться на L2. Для связи нужен маршрутизатор:
```
         Коммутатор
        ┌──────────┐
VLAN10──┤          ├──── Транк ────► Маршрутизатор (sub-интерфейсы)
VLAN20──┤          │                  eth0.10 → 192.168.10.1
VLAN30──┤          │                  eth0.20 → 192.168.20.1
        └──────────┘
```
Либо используется **L3-коммутатор** (Switch Virtual Interface, SVI) — маршрутизация прямо на коммутаторе.

## 7. STP — протокол остовного дерева (IEEE 802.1D)
> **STP (Spanning Tree Protocol)** — протокол, предотвращающий **петли** в сети с резервными соединениями. Логически блокирует избыточные порты, оставляя одно активное дерево связей.
## 7.1.  Проблема: петли на L2
В отличие от IP (TTL), фреймы Ethernet не имеют счётчика жизни. Broadcast-фрейм в петле будет ходить вечно — **broadcast storm**, коллапс сети.
```
     [SW1]────────[SW2]
       │              │
       └────[SW3]─────┘

Broadcast от SW1 → SW2 → SW3 → SW1 → SW2 → ... ∞
```
## 7.2. Как работает STP
**Шаг 1 — Выбор Root Bridge**  
Коммутатор с наименьшим **Bridge ID** становится корнем дерева.  
`Bridge ID = Priority (2 байт) + MAC-адрес (6 байт)`. По умолчанию priority = 32768.

**Шаг 2 — Выбор Root Port (RP)**  
На каждом некорневом коммутаторе — порт с наименьшей стоимостью пути до Root Bridge.

**Шаг 3 — Выбор Designated Port (DP)**  
На каждом сегменте — один порт пересылает трафик в сторону Root Bridge.

**Шаг 4 — Блокировка остальных портов**  
Все порты, не ставшие RP или DP, переходят в **Blocking** — петля разорвана.

## 7.3. Стоимость портов (Path Cost)

| Скорость | Стоимость (802.1D) |
|----------|-------------------|
| 10 Мбит/с | 100 |
| 100 Мбит/с | 19 |
| 1 Гбит/с | 4 |
| 10 Гбит/с | 2 |
## 7.4. Состояния портов STP

```
Blocking → Listening → Learning → Forwarding
  (15 сек)   (15 сек)
```

| Состояние | Получает BPDU | Отправляет BPDU | Учит MAC | Пересылает фреймы |
|-----------|--------------|-----------------|----------|------------------|
| **Blocking** | ✓ | ✗ | ✗ | ✗ |
| **Listening** | ✓ | ✓ | ✗ | ✗ |
| **Learning** | ✓ | ✓ | ✓ | ✗ |
| **Forwarding** | ✓ | ✓ | ✓ | ✓ |
| **Disabled** | ✗ | ✗ | ✗ | ✗ |

> Сходимость классического STP при падении линка: **30–50 секунд** — критически долго для современных сетей.

## 7.5. BPDU — Bridge Protocol Data Unit

Служебные фреймы, которыми коммутаторы обмениваются для построения дерева. Содержат Bridge ID, стоимость пути, таймеры.
## 7.6. Эволюция STP

| Протокол | Стандарт | Сходимость | Особенности |
|----------|----------|------------|-------------|
| **STP** | 802.1D | 30–50 сек | Оригинал |
| **RSTP** | 802.1w | < 1 сек | Rapid STP, новые состояния портов, быстрее сходится |
| **MSTP** | 802.1s | < 1 сек | Multiple STP, разные деревья для разных VLAN |
| **PVST+** | Cisco | < 1 сек | Per-VLAN STP, отдельное дерево на каждый VLAN (проприетарный) |
## 7.7. RSTP — ключевые улучшения

- Новые роли портов: **Alternate** (резервный RP) и **Backup** (резервный DP)
- **Proposal/Agreement** механизм — быстрый переход без ожидания таймеров
- **Edge Port** (аналог PortFast) — порты к конечным устройствам сразу в Forwarding

> [!NOTE] PortFast / Edge Port
> Порты к конечным устройствам (ПК, серверам) не должны участвовать в STP-вычислениях. `PortFast` (Cisco) / Edge Port (RSTP) переводит порт сразу в Forwarding, минуя Listening и Learning.

# 9. Пакет после выхода из шлюза
Между отправителем и получателем работают **два разных механизма одновременно**:
```
Твой ПК → Шлюз → [Интернет] → Сервер

         L3 (IP): знает конечный IP-адрес назначения
         ────────────────────────────────────────────►
         
         L2 (Ethernet): знает только MAC следующего прыжка
         [ПК→Шлюз] [Шлюз→Роутер1] [Роутер1→Роутер2] ...
```
На каждом **прыжке (hop)** происходит одно и то же:

**1.** Роутер смотрит в IP-заголовок — куда в итоге надо доставить пакет.  
**2.** По таблице маршрутизации определяет — через какой **следующий роутер** (next hop) отправить.  
**3.** Делает ARP-запрос: _«кто такой 10.0.0.1?»_ — и узнаёт его MAC.  
**4.** Оборачивает пакет в **новый Ethernet-фрейм** с MAC следующего роутера и отправляет.  
**5.** Следующий роутер **снимает** Ethernet-фрейм, смотрит IP, повторяет с шага 1.

```
Пакет в пути:

Прыжок 1:  [MAC_шлюза  | MAC_твоего_ПК | IP_сервера | IP_тебя | данные]
                             ↓ шлюз меняет MAC-обёртку
Прыжок 2:  [MAC_роутера2 | MAC_шлюза   | IP_сервера | IP_тебя | данные]
                             ↓ роутер2 меняет MAC-обёртку
Прыжок 3:  [MAC_сервера | MAC_роутера2 | IP_сервера | IP_тебя | данные]
```

IP-заголовок при этом **не меняется** от начала до конца (если нет [[NAT|NAT]]). Меняется только Ethernet-обёртка на каждом прыжке.

По сути Ethernet — это просто способ доставить пакет **до следующего роутера**, не до конечного адреса. Конечный IP знает только L3, а Ethernet об этом вообще не в курсе.
