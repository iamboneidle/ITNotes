# 1. Что это?
gRPC — это фреймворк, который берет HTTP/2 в качестве транспорта и **Protocol Buffers (Protobuf)** в качестве формата данных.

**Почему gRPC — это не REST?**
1. **Контракт (IDL):** Вы сначала описываете `.proto` файл. Это «закон». Из него генерируется код для клиента и сервера на любом языке.
2. **Бинарный формат:** Protobuf сериализует данные в компактные бинарные блоки. Это гораздо меньше и быстрее, чем JSON.
3. **Типы взаимодействия:**
    - **Unary:** Классический «запрос-ответ».
    - **Server Streaming:** Клиент шлет запрос, сервер шлет поток данных (например, лента новостей).
    - **Client Streaming:** Клиент шлет поток данных (загрузка файла), сервер отвечает один раз.
    - **Bidirectional Streaming:** Оба шлют данные одновременно (чаты, real-time игры).

# 2. HTTP/2 HOL Blocking
Если в TCP-соединении потеряется хотя бы один пакет, всё соединение встанет на паузу (TCP Window замрет), пока потеря не будет восстановлена. В этот момент **все** мультиплексированные стримы внутри этого соединения будут ждать. Именно поэтому появился **HTTP/3 (QUIC)** ([[HTTP-3|link]]), работающий поверх UDP, где потеря пакета в одном стриме не влияет на остальные.

# 3. gRPC и балансировка (L4 vs L7)
Поскольку gRPC держит одно долгоживущее TCP-соединение, обычные балансировщики L4 (например, облачные NLB или классический сервис Kubernetes в режиме iptables) не могут распределять запросы.
- **Проблема:** Клиент подключился к Поду №1. Все последующие тысячи gRPC-вызовов пойдут в Под №1, даже если рядом появились еще 10 подов.
- **Решение:** Балансировка на уровне L7.
    - **Service Mesh (Linkerd/Istio):** Sidecar-прокси перехватывает gRPC-вызовы и балансирует каждый отдельный _стрим_.
    - **Client-side load balancing:** Клиент получает список всех IP подов и сам распределяет запросы между ними.
    - **Headless Services:** В K8s используется для получения списка всех IP через DNS.

# 4. Remote Protocol Call
## 4.1. Stub & Skeleton
Чтобы вызов выглядел как «родной», RPC использует два посредника:
1. **Stub (Клиентская заглушка):** Это локальный объект в коде клиента, который имитирует целевую функцию. Когда ты вызываешь `service.GetUser(id)`, ты вызываешь метод Стаба.
2. **Skeleton (Серверный скелет):** Это прослойка на стороне сервера, которая принимает сетевой запрос, распаковывает его и вызывает реальный код функции в приложении.
## 4.2. Пошаговый путь вызова
### 4.2.1. Сериализация
Стаб берет аргументы функции (например, строку и число) и превращает их в бинарный формат.
- В **gRPC** это делает **Protobuf**.
- В старых системах это мог быть JSON или XML. Бинарный формат важен, потому что разные языки по-разному хранят данные в памяти (например, порядок байтов — _Endianness_).
### 4.2.2. Передача
Стаб передает этот бинарный блок транспортному уровню (обычно **HTTP/2** для gRPC или чистый **TCP**). Ядро ОС упаковывает это в пакеты и отправляет через сеть.
### 4.2.3. Демоmarshalling и Выполнение
Скелет на сервере получает пакеты, собирает бинарный блок обратно и превращает его в объекты, понятные языку программирования сервера (Go, Java, Python). Затем он вызывает реальную функцию.
### 4.2.4. Обратный путь
Результат выполнения функции (например, объект пользователя) проходит тот же путь: Сервер -> Скелет (Marshalling) -> Сеть -> Стаб (Unmarshalling) -> Клиент.
## 4.3. IDL
В REST ты можешь отправить JSON с опечаткой в поле, и сервер может просто проигнорировать его. В RPC так нельзя. Чтобы клиент и сервер понимали друг друга, используется **IDL (Interface Definition Language)**.

В gRPC это `.proto` файл:
```Protocol Buffers
service UserService {
	rpc GetUser (UserRequest) returns (UserResponse); 
}
```
Из этого файла специальный компилятор (`protoc`) генерирует код Стаба и Скелета на нужных языках. Это гарантирует **строгую типизацию**.
