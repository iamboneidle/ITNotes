# 1. Кластерная архитектура
Kafka работает в виде **кластера**, который состоит из одного или нескольких серверов.
- **Brokers (Брокеры):** Это отдельные узлы (серверы) кластера. Каждый брокер отвечает за хранение части данных и обработку запросов от клиентов (производителей и потребителей). Брокеры работают сообща, обеспечивая отказоустойчивость.
- **Controllers (Контроллеры):** В кластере один из брокеров всегда выполняет роль контроллера. Он — «мозг» кластера. Контроллер следит за состоянием других брокеров, управляет списком разделов и назначает лидеров для каждой партиции.
    - _Важно:_ Раньше для этого требовался внешний сервис **Zookeeper**, но в современных версиях Kafka (с протоколом KRaft) функции контроллера встроены прямо в брокеры.

# 2. Логическая структура
Данные в Kafka не сваливаются в одну кучу, а строго структурированы.
- **Topics (Топики):** Это логическая категория или имя канала, куда отправляются сообщения. Например, топик `user-clicks` или `order-transactions`. Можно сравнить топик с таблицей в БД, но без жесткой схемы.
- **Partitions (Партиции/Разделы):** Топик физически разделен на несколько частей — партиций.
    - **Масштабируемость:** Партиции позволяют распределить данные одного топика между разными брокерами. Это дает возможность записывать и читать данные параллельно.
    - **Порядок:** Внутри одной партиции сообщения строго упорядочены по времени их поступления. Однако порядок во всем топике (между разными партициями) не гарантируется.
    - **Оффсет (Offset):** Каждое сообщение в партиции получает уникальный порядковый номер — оффсет. Это позволяет потребителю знать, на каком месте он остановился.
___
# 3. Репликация и отказоустойчивость
Kafka гарантирует, что данные не пропадут, даже если один из брокеров выйдет из строя.
- **Replication Factor (Фактор репликации):** Обычно устанавливается значение 3. Это значит, что каждая партиция копируется на 3 разных брокера.
- **Leader & Followers:**
    - **Leader:** Одна из копий партиции назначается лидером. Все записи и чтения идут только через него.
    - **Follower:** Остальные копии (фолловеры) просто пассивно копируют данные лидера. Если лидер падает, контроллер выбирает одного из фолловеров и назначает его новым лидером.
___
# 4. Как это работает вместе
1. **Producer (Производитель):** Отправляет сообщение в топик. Он сам решает, в какую партицию попадет сообщение (обычно по хэшу ключа, чтобы сообщения с одним ID всегда попадали в одну партицию).
2. **Broker:** Принимает сообщение, записывает его в конец лога на диске и подтверждает запись.
3. **Consumer (Потребитель):** Подписывается на топик. Он читает сообщения из партиций, двигая свой «курсор» (оффсет). Несколько потребителей могут объединяться в **Consumer Group**, чтобы делить партиции между собой и обрабатывать данные параллельно.
