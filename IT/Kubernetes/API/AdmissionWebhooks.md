# 1. Жизненный путь запроса
Когда ты делаешь `kubectl apply -f pod.yaml`, запрос проходит через три фазы контроллеров:

1. **Mutating (Изменяющие):** Могут модифицировать объект. Например, добавить `sidecar-контейнер` (Istio/Linkerd) или проставить дефолтные лимиты ресурсов.
2. **Object Schema Validation:** Проверка того, что JSON/YAML соответствует схеме Kubernetes.
3. **Validating (Проверяющие):** Только одобряют или отклоняют запрос. Например, проверяют, что образ скачивается только из доверенного Registry.

**Важное правило:** Сначала всегда отрабатывают все Mutating контроллеры, и только потом — Validating. Если хотя бы один контроллер на любом этапе отклонит запрос, пользователь получит ошибку, и объект не будет создан.
___
# 2. Специальные типы
Большинство базовых контроллеров встроены в бинарник `kube-apiserver` (например, `ServiceAccount` или `NodeRestriction`), но для кастомной логики используются **Dynamic Admission Control** через вебхуки:

- **MutatingAdmissionWebhook:** API Server отправляет JSON с объектом на твой внешний сервис. Сервис возвращает **JSON Patch** (набор инструкций, что изменить).
- **ValidatingAdmissionWebhook:** Аналогично, но сервис возвращает только `allowed: true/false` и сообщение об ошибке.
> [!example] 
> Как работает **Istio**? Он регистрирует `MutatingAdmissionWebhook`. Когда ты создаешь под в неймспейсе с меткой `istio-injection=enabled`, контроллер перехватывает запрос и «подклеивает» в описание пода описание контейнера `istio-proxy`.
