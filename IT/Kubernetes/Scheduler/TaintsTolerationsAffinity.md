# 1. Taints & Tolerations
Эта связка работает по принципу **отталкивания**. Taint (отметка) вешается на **Ноду**, а Toleration (допуск) — на **Под**.
**Taints (На ноде)**
Когда ты вешаешь таинт, ты говоришь: «Эта нода проблемная/особенная, не пускай сюда никого, кто не знает, как с этим жить».
- **Эффекты:**
    - `NoSchedule`: Новые поды не придут. Старые остаются.
    - `PreferNoSchedule`: Мягкий запрет (для стадии Scoring).
    - `NoExecute`: Выселяет уже запущенные поды, если у них нет допуска.
**Tolerations (На поде)**
Это «пропуск», позволяющий поду игнорировать таинт.
- **Важный нюанс:** Наличие Toleration **не гарантирует**, что под попадет именно на эту ноду. Он просто _может_ на ней находиться. Чтобы заставить его туда пойти, нужно Affinity.
## 1.1. Вешаем таинт на ноды
```shell
kubectl taint nodes node-db-01 app=database:NoSchedule
```
Здесь `app=database` — это пара key=value, а `NoSchedule` — эффект.
**Теперь обычные поды туда не попадут.** Чтобы база данных там запустилась, в её манифесте (StatefulSet) должен быть допуск:
```yaml
spec:
  tolerations:
  - key: "app"
    operator: "Equal"
    value: "database"
    effect: "NoSchedule"
```
# 1.2. Ноды с GPU
Вешаем таинт:
```shell
kubectl taint nodes node-gpu-01 hardware=gpu:NoSchedule
```
Используем `NoExecute` для очистки:
```shell
kubectl taint nodes node-gpu-01 hardware=gpu:NoExecute
```
Все поды без допуска мгновенно «улетят» с ноды на другие свободные узлы.
## 1.3. Тонкий момент: Операторы `Exists` vs `Equal`
- **`Equal`**: Требует совпадения и ключа, и значения.
    - _Пример:_ `key: "app", value: "db", operator: "Equal"` — сработает только на таинт `app=db`.
- **`Exists`**: Проверяет только наличие ключа. Это удобно, когда у тебя много нод с разными значениями одного таинта.
    - _Пример:_ `key: "app", operator: "Exists"` — сработает на `app=db`, `app=web`, `app=cache`.

# 2. Node Affinity
Это развитие концепции `nodeSelector`. Позволяет использовать логические операторы (`In`, `NotIn`, `Exists`, `DoesNotExist`, `Gt`, `Lt`).
**Типы Affinity:**
1. **`requiredDuringSchedulingIgnoredDuringExecution` (Hard):** Жесткое правило. Если подходящей ноды нет — под будет висеть в `Pending`.
2. **`preferredDuringSchedulingIgnoredDuringExecution` (Soft):** «Пожелание». Планировщик начислит ноде дополнительные баллы (weight от 1 до 100), если она соответствует условию.
> **Почему "IgnoredDuringExecution"?** Если ты изменишь метку на ноде, когда под уже запущен, Kubernetes не будет его убивать. Он проверяет условия только в момент выбора ноды.
# 3. Pod Affinity & Anti-Affinity
Это самый сложный и ресурсозатратный механизм планировщика, так как он заставляет его проверять не только метки нод, но и метки **всех уже запущенных подов**.

- **Pod Affinity:** «Запусти меня там, где уже живет Под X» (например, кеш рядом с приложением).
- **Pod Anti-Affinity:** «Не запускай меня там, где уже живет Под X» (например, не класть две реплики одной БД на одну ноду для отказоустойчивости).

**TopologyKey**
Это ключ, определяющий границы «соседства».
- `kubernetes.io/hostname`: Соседство в рамках одной железки.
- `topology.kubernetes.io/zone`: Соседство в рамках одного дата-центра/зоны.
