# 1. Что такое Ingress
> [!info] 
> _Ingress_ (существительное) – это вход или вхождение, право на вход, средство или место входа, проход. 

Одна из важных причин заключается в том, что для каждой службы LoadBalancer требуется собственный балансировщик нагрузки с собственным общедоступным IP-адресом, в то время как для Ingress’а требуется только один, даже когда предоставляется доступ к десяткам служб. Когда клиент отправляет HTTP-запрос ко входу, хост и путь в запросе определяют, к какой службе этот запрос перенаправляется.
![[IngressInAction]]
Ingress’ы оперируют на уровне приложений сетевого стека (HTTP) и могут предоставлять такой функционал, как сохранение сессий на основе файлов
cookie и т.п., чего не могут делать службы.

# 2. Создание ресурса Ingress
```yaml
apiVersion: extensions/v1beta1
kind: Ingress
metadata:
  name: kubia
spec:
  rules:
    – host: kubia.example.com # |<- увязывает это доменное со службой Ingress
      http:
        paths:
          – path: /                       # |<-
            backend:                      # | Все запросы будут отправляться на
              serviceName: kubia-nodeport # | порт 80 службы kubia-nodeport
              servicePort: 80             # |__
```
Это определение задает ресурс Ingress с единственным правилом, которое удостоверяется, что все HTTP-запросы, полученные контроллером входа, в котором запрашивает хост `kubia.example.com`, будут отправлены в службу `kubia-nodeport` на порту `80`.
> [!important] 
> От контроллеров входа на облачных провайдерах (например, в GKE) требуется, чтобы Ingress указывал на службу NodePort. Но это не является
требованием самого Kubernetes. 

# 3. Доступ к службе через Ingress
Для того чтобы получить доступ к службе через `http://kubia.example.com`, вы должны убедиться, что доменное имя привязано к IP-адресу контроллера Ingress.
## 3.1. Получение IP-адреса входа
Для того чтобы взглянуть на IP-адрес, вам нужно вывести список Ingress’ов:
```shell
kubectl get ingresses
```
output:
```
NAME  HOSTS             ADDRESS        PORTS AGE
kubia kubia.example.com 192.168.99.100 80    29m
```
## 3.2. Обеспечение соответствия имени хоста, назначенного в Ingress, IP-адресу Ingress
После того как вы узнали IP-адрес, можно настроить DNS-серверы для привязки `kubia.example.com` к этому IP-адресу либо добавить в `/etc/hosts` (или в
`C:\windows\system32\drivers\etc\hosts в Windows`) следующую строку:
```
192.168.99.100 kubia.example.com
```
## 3.3. Доступ к модулям через Ingress
Теперь все настроено, поэтому доступ к службе можно получить по адресу
`http://kubia.example.com` (используя браузер или `curl`):
```shell
curl http://kubia.example.com
```

# 4. Как работает Ingress
Клиент сначала выполнил DNS-поиск kubia.example.com, и DNS-сервер (или локальная операционная система) вернул IP-адрес контроллера Ingress. Затем клиент отправил HTTP-запрос контроллеру Ingress и в заголовке Host указал kubia.example.com. Из этого заголовка контроллер определил, к какой службе клиент пытается обратиться, отыскал IP-адреса модулей через связанный со службой объект Endpoints и перенаправил запрос клиента одному из модулей.

# 5. Обеспечение доступа ко множеству служб через один и тот же Ingress
## 5.1. Увязывание разных служб с разными путями одного хоста
```yaml
...
– host: kubia.example.com
  http:
    paths:
      – path: /kubia
        backend:             # |<- Запросы к kubia.example.com/kubia
          serviceName: kubia # | будут перенаправлены в службу kubia
            servicePort: 80  # |__
      – path: /foo
        backend:             # |<- Запросы к kubia.example.com/bar
          serviceName: bar   # | будут перенаправлены в службу bar
            servicePort: 80  # |__
```
В данном случае запросы будут отправлены в две разные службы в зависимости от пути в запрашиваемом URL-адресе. Поэтому клиенты могут достичь 2 разных служб через единый IP-адрес (адрес контроллера Ingress).
## 5.2. Увязывание разных служб с разными узлами
Аналогичным образом Ingress можно использовать для увязывания разных служб на основе хоста в HTTP-запросе, а не (только) пути:
```yaml
spec:
  rules:
    – host: foo.example.com    # |<-
      http:                    # |
        paths:                 # | Запросы к foo.example.com будут 
          – path: /            # | перенаправлены в службу foo
            backend:           # |
              serviceName: foo # |
              servicePort: 80  # |__
    – host: bar.example.com    # |<-
      http:                    # |
        paths:                 # | Запросы к bar.example.com будут
          – path: /            # | перенаправлены в службу bar
            backend:           # |
              serviceName: bar # |
              servicePort: 80  # |__
```
