# 1. Как CoreDNS узнает о новых сервисах?
Внутри CoreDNS работает плагин **kubernetes**. Он подключается к API-серверу и подписывается на события создания/удаления `Services` и `Endpoints`.

Когда ты создаешь сервис `my-svc` в неймспейсе `prod`, CoreDNS мгновенно формирует запись: `my-svc.prod.svc.cluster.local` -> `ClusterIP`.

Если это **Headless Service** (с `clusterIP: None`), CoreDNS отдаёт не один IP, а список IP всех подов, которые стоят за этим сервисом.

# 2. Анатомия Corefile
Вся логика CoreDNS описана в `ConfigMap` (обычно в неймспейсе `kube-system`). Это цепочка плагинов, через которые пакет проходит сверху вниз:
```
.:53 {
    errors          # Логирование ошибок
    health          # Liveness probe для пода самого CoreDNS
    ready           # Readiness probe
    kubernetes cluster.local in-addr.arpa ip6.arpa {
       pods insecure
       fallthrough in-addr.arpa ip6.arpa
    }
    prometheus :9153 # Метрики (тот самый трафик, за которым надо следить)
    forward . /etc/resolv.conf # Если не нашли внутри, идем в апстрим (на ноду)
    cache 30        # Кэширование ответов на 30 секунд
    loop            # Защита от бесконечных циклов резолвинга
    reload          # Позволяет менять конфиг без рестарта пода
}
```

# 3. Проблемы производительности и масштабирование
Поскольку CoreDNS — это обычный под, он может стать "узким местом".
## 3.1. Масштабирование (Autoscaling)
В больших кластерах используется **cluster-proportional-autoscaler**. Он увеличивает количество реплик CoreDNS в зависимости от количества ядер или нод в кластере. 
### 3.2. Мониторинг "битых" запросов
Если CoreDNS начинает тормозить, смотри на метрику: `coredns_dns_responses_total{rcode="NXDOMAIN"}` Если количество `NXDOMAIN` зашкаливает — значит, у тебя либо криво настроены приложения (ищут то, чего нет), либо те самые `ndots` заставляют систему генерировать тонны мусорных запросов.

# 4. NodeLocal DNSCache
Это решение, которое де-факто стало стандартом для нагруженных кластеров.
- **Как было:** Поды со всего кластера бегут в центральные 2-3 пода CoreDNS через `iptables/conntrack` (отсюда те самые задержки и Race Conditions).
- **Как стало:** На каждой ноде запускается мелкий кэширующий агент. Под идет на `169.254.25.10`.
- **Профит:**
    1. Резко падает нагрузка на центральный CoreDNS.
    2. Исчезают Race Conditions в `conntrack`, так как запросы идут локально.
    3. Время ответа падает в разы.
