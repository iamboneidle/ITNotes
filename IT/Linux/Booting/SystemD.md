# 1. О системах инициализации
В операционной системе Linux и других системах семейства Unix после завершения загрузки ядра начинается инициализация Linux системы, сервисов и других компонентов. За это отвечает процесс инициализации, он запускается ядром сразу после завершения загрузки, имеет PID 1, и будет выполняться пока будет работать система.
## 1.1. Задачи системы инициализации
1. **Запуск всей системы** — последовательный или параллельный старт всех необходимых сервисов (логин-менеджер, сеть, веб-сервер, GUI).
2. **Мониторинг процессов** — если какой-то критический сервис упал, система инициализации может перезапустить его.
3. **Управление уровнями запуска (runlevels)** — позволяет переключать режимы работы (обычная работа, восстановление, выключение, однопользовательский режим).
4. **Корректное завершение работы** — при выключении компьютера init останавливает все службы в правильном порядке, чтобы не потерять данные.

Без init система после загрузки ядра просто выдаст ошибку `Kernel panic — no init found`.
## 1.2. Системы инициализации

| Система           | Где используется                                                                               | Особенность                                                                                                                                                                                 |
| ----------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **System V init** | Устаревшие дистрибутивы (старые Slackware, RHEL 5, Debian до 6)                                | Последовательный запуск скриптов. Уровни запуска (runlevels 0–6). Медленно, но очень просто и понятно.                                                                                      |
| **systemd**       | **Почти все современные дистрибутивы** (Ubuntu, Fedora, RHEL/CentOS 7+, Debian 8+, Arch Linux) | Параллельный запуск сервисов (ускорение), отслеживание зависимостей, собственный журнал логов (journald), управление сокетами, таймерами (замена cron). Самый распространённый.             |
| **Upstart**       | Ubuntu с 6.10 по 14.10, некоторые старые версии Chrome OS                                      | Событийно-ориентированный: реагирует не на уровни запуска, а на события (например, «сеть появилась» → запустить веб-сервер). Умел перезапускать упавшие сервисы. Сейчас заменён на systemd. |
| **OpenRC**        | Gentoo, Alpine Linux, частично в системах без systemd                                          | Надстройка над SysV init, но с параллельным запуском и модульностью. Ценится за простоту и предсказуемость.                                                                                 |
| **runit**         | Void Linux, некоторые контейнеры                                                               | Очень минималистичный, постоянно следит за процессами и автоматически их перезапускает. Популярен во встраиваемых системах.                                                                 |
# 2. Как понять, используется ли systemd
`/run/systemd/` говорит о наличии systemd в системе.
`/run/systemd/system/` - говорит о том, что это активная система инициализации
Можно через stat узнать
Если есть симлинк - используется systemd.
```
stat /sbin/initFile: ‘/sbin/init’ -> ‘../lib/systemd/systemd’
```
Можно сделать ещё через процесс 1, и через файловую систему `/proc`:
```
root@swfuse:~# stat /proc/1/exe

Device: 4h/4d	Inode: 444358      Links: 1
Access: (0777/lrwxrwxrwx)  Uid: (    0/    root)   Gid: (    0/    root)
Access: 2022-06-20 00:00:01.579875809 +0000
Modify: 2022-06-09 17:19:41.581240179 +0000
Change: 2022-06-09 17:19:41.581240179 +0000
```

# 3. Таргеты и run-levels
`systemd target` — это специальный юнит (файл с расширением `.target`), который служит точкой синхронизации или группировки других юнитов. В отличие от сервисов, таргеты не выполняют конкретный код, а определяют **состояние системы**.
## 3.1. Основные концепции
- **Группировка:** Таргет собирает зависимости (сервисы, сокеты, другие таргеты) через механизмы `Wants` и `Requires`.
- **Иерархия:** Таргеты могут наследоваться. Например, `graphical.target` включает в себя `multi-user.target`.
- **Замена Runlevels:** В классическом `init` уровни выполнения были жестко заданы цифрами (0-6). В `systemd` это гибкие объекты с понятными именами.
> [!note] 
> *Requires*:
> - **При запуске:** Если запустить **А**, `systemd` одновременно попытается запустить **Б**. Если **Б** не сможет запуститься (ошибка в конфиге, отсутствие бинарника), то **А** тоже **не будет запущен**.
> - **В процессе работы:** Если **Б** внезапно упадет или будет остановлен вручную, юнит **А** будет **автоматически остановлен** вслед за ним.
>
>*Wants*:
>- **При запуске:** Если запустить **А**, `systemd` также попытается запустить **Б**. Однако, если **Б** не сможет запуститься (вылетит с ошибкой), это **никак не повлияет на запуск А**. Юнит **А** продолжит работу.
>- **В процессе работы:** Если **Б** упадет или будет остановлен, юнит **А** **продолжит работать** как ни в чем не бывало.
## 3.2. Соответствие Runlevels и Targets

| **Runlevel** | **Target**          | **Описание**                                               |
| ------------ | ------------------- | ---------------------------------------------------------- |
| **0**        | `poweroff.target`   | Выключение системы.                                        |
| **1**        | `rescue.target`     | Однопользовательский режим (root shell, минимум сервисов). |
| **2, 3, 4**  | `multi-user.target` | Многопользовательский режим с сетью, без графики.          |
| **5**        | `graphical.target`  | Многопользовательский режим с сетью и GUI.                 |
| **6**        | `reboot.target`     | Перезагрузка системы.                                      |
## 3.3. Таргеты, которые есть всегда (Основные)
- **`default.target`**: Символическая ссылка на таргет, который загружается по умолчанию (обычно указывает на `multi-user` или `graphical`).
- **`basic.target`**: Базовая инициализация (монтирование ФС, инициализация сокетов, драйверов).
- **`sysinit.target`**: Ранняя стадия загрузки (критически важные системные службы).
- **`emergency.target`**: Самый минимальный режим (только консоль root, ФС может быть в read-only). Используется для "оживления" упавшей системы.
- **`timers.target`**: Группа для всех активных таймеров в системе.
## 3.4. Управление и переключение
Для работы с таргетами используется утилита `systemctl`.
### 3.4.1. Просмотр состояния
**Текущий таргет по умолчанию:**
```shell
systemctl get-default
```
**Список всех загруженных таргетов:**
```shell
systemctl list-units --type=target
```
### 3.4.2. Смена таргета (на лету)
Это аналог команды `init 3` или `init 5`. В `systemd` используется команда `isolate`, которая останавливает всё, что не входит в состав нового таргета.
**Перейти в консольный режим:**
```shell
systemctl isolate multi-user.target
```
**Перейти в графический режим:**
```shell
systemctl isolate graphical.target
```
### 3.4.3. Изменение режима загрузки (на постоянку)
**Сделать систему серверной (без GUI при загрузке):**
```shell
systemctl set-default multi-user.target
```
**Вернуть графический интерфейс при старте:**
```shell
systemctl set-default graphical.target
```
## 3.5. Полезные команды для отладки
Чтобы посмотреть дерево зависимостей конкретного таргета и понять, почему он долго грузится или что в него входит:
```shell
systemctl list-dependencies multi-user.target
```
Для визуализации критического пути загрузки (что за чем шло):
```shell
systemd-analyze plot > boot.svg
```
