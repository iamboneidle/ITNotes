# 1. Что это?
**Conntrack (Connection Tracking)** — это модуль ядра Linux (часть подсистемы Netfilter), который позволяет операционной системе «помнить» о состоянии сетевых сессий. Без него Linux был бы «глупым» пакетным фильтром (Stateless), который рассматривает каждый пакет в отрыве от остальных.
что Conntrack — это база для работы **NAT** и **Stateful Firewall** (например, правил `iptables` с ключом `-m state --state ESTABLISHED`).
___
# 2. Как это работает
Когда пакет попадает в сетевой стек, conntrack пытается сопоставить его с уже существующей записью в своей таблице.

**Структура записи (Tuple)**
Соединение идентифицируется через **Tuple** (кортеж), который состоит из:
- Protocol (TCP, UDP, ICMP)
- Source IP / Source Port
- Destination IP / Destination Port
Для каждого соединения в таблице хранятся два таких кортежа: оригинальный (от клиента к серверу) и ответный (от сервера к клиенту). Это позволяет ядру понимать, что входящий пакет от сервера — это на самом деле ответ на запрос нашего клиента.

**Состояния (States)**
Conntrack вводит свои состояния, которые не всегда совпадают с состояниями TCP:
1. **NEW**: Первый пакет сессии (например, TCP SYN).
2. **ESTABLISHED**: Ответный пакет получен, двусторонний обмен подтвержден.
3. **RELATED**: Пакет, порожденный существующим соединением (например, ICMP error или FTP data connection).
4. **INVALID**: Пакет, который не удалось идентифицировать (битый или вне окна).
___
# 3. Почему таблица переполняется
Таблица conntrack — это хеш-таблица в оперативной памяти с фиксированным максимальным размером.

**Основные причины переполнения:**
1. **Высокий PPS (Packets Per Second) / High Concurrency**: Огромное количество коротких соединений (например, микросервисы без Keep-Alive или DNS-запросы).
2. **DDoS-атаки**: Типичный SYN-flood забивает таблицу записями в состоянии `SYN_RECV`, которые висят там до тайм-аута.
3. **Неоптимальные тайм-ауты**: По умолчанию Linux может хранить запись о закрытом TCP-соединении (TIME_WAIT) довольно долго, занимая драгоценный слот.

**Что происходит при переполнении?**
Когда лимит достигнут, ядро начинает **дропать (отбрасывать)** все новые пакеты, которые не относятся к уже установленным соединениям. В логах `dmesg` появляется фатальная надпись: `nf_conntrack: table full, dropping packet`. Для пользователя это выглядит как полная недоступность сервиса («сеть лежит»), хотя CPU и RAM могут быть свободны.

# 4. Мониторинг и диагностика
Если ты подозреваешь проблемы с conntrack:
- **Проверить текущее количество записей:** `sysctl net.netfilter.nf_conntrack_count`
- **Узнать лимит:** `sysctl net.netfilter.nf_conntrack_max`
- **Посмотреть саму таблицу (нужна утилита conntrack):** `conntrack -L` (позволяет увидеть, какие именно IP «съели» таблицу).

# 5. Тюнинг и решение проблемы
## 5.1. Увеличить лимиты
Самый простой, но «прожорливый» путь
```shell
sysctl -w net.netfilter.nf_conntrack_max=1048576
```
*(Важно также пропорционально увеличивать `net.netfilter.nf_conntrack_buckets`, чтобы хеш-таблица не стала слишком глубокой, иначе вырастет нагрузка на CPU при поиске).*
## 5.2. Уменьшить тайм-ауты
Заставить ядро быстрее забывать о неактивных соединениях:
```shell
sysctl -w net.netfilter.nf_conntrack_tcp_timeout_established=600
sysctl -w net.netfilter.nf_conntrack_tcp_timeout_time_wait=30
```
## 5.3. Использование NOTRACK
Если у тебя есть высоконагруженный сервис (например, балансировщик или DNS), трафик которого не требует NAT или фильтрации по состояниям, его можно исключить из отслеживания:
```shell
iptables -t raw -A PREROUTING -p tcp --dport 80 -j NOTRACK
```
*Пакеты на порт 80 пролетят мимо таблицы conntrack, не занимая в ней места.*
