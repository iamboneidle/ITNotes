# 1. Физика процесса
Когда процесс запрашивает данные с диска, он переходит в состояние **Uninterruptible Sleep (D)**. Планировщик Linux видит, что ядро CPU освободилось, но если в очереди больше нет готовых к выполнению задач (Runable), он помечает это время как `%iowait`.

Если у тебя `%wa` выше 10-15%, это сигнализирует о том, что диски не справляются с потоком данных.

# 2. Алгоритм дебага: От общего к частному
## 2.1. Подтверждаем проблему (iostat)
Утилита `iostat` из пакета `sysstat`:
```shell
iostat -xz 1
```
Смотри на колонки:

- **%util:** Процент времени, когда диск был занят работой. Если тут 90-100%, диск «забит».    
- **await:** Среднее время (в мс) ответа на запрос. Для SSD норма < 1-2мс, для HDD — до 10-20мс. Если видишь 100мс+ — всё плохо.
- **aqu-sz:** Размер очереди запросов. Если очередь растет, диск не успевает переваривать запросы.
## 2.2. Ищем «виновника» (iotop)
`iotop` покажет, какой конкретно процесс сейчас генерирует нагрузку.
```shell
iotop -oP
```
Флаг `-o` покажет только активные процессы, `-P` — только процессы (без потоков). Ты сразу увидишь, кто делает `DISK WRITE` или `DISK READ`.
## 2.3. Глубокий анализ (lsof / strace)
Допустим, ты нашел PID процесса. Теперь надо понять, что именно он пишет.
```shell
lsof -p <PID> | grep REG
```
Это покажет открытые файлы. Если файлов слишком много или это огромные логи, причина ясна.
Если нужно понять характер записи (синхронная/асинхронная), используй:
```shell
strace -p <PID> -e trace=read,write,fsync
```
Если процесс постоянно делает `fsync()`, он заставляет ядро сбрасывать кэш на диск после каждой записи, что дико тормозит систему.

# 3. Пути решения
## 3.1. Оптимизация на уровне ОС (Sysctl)
Linux использует **Page Cache** для кэширования дисковых операций. Можно подкрутить параметры «грязных страниц» (dirty pages):
- `vm.dirty_ratio`: Процент памяти, при котором данные начнут принудительно сбрасываться на диск.
- `vm.dirty_background_ratio`: Процент, при котором фоновое сбрасывание начнется раньше.
- **Увеличение этих значений** поможет сгладить пики записи, но повысит риск потери данных при выключении питания.
## 3.2. Оптимизация планировщика I/O (I/O Schedulers)
Проверь, какой планировщик используется: `cat /sys/block/sdX/queue/scheduler`.
- **Deadline / MQ-Deadline:** Хорош для SSD и баз данных (минимизирует задержки).
- **Kyber:** Современный планировщик для быстрых NVMe.
- **BFQ:** Хорош для десктопов или медленных HDD (делает систему «отзывчивой»).
## 3.3. Архитектурные правки
- **Noatime:** Монтируй разделы с флагом `noatime`. По умолчанию Linux обновляет время последнего доступа к файлу при каждом чтении, что создает лишнюю запись.
- **Separate Disks:** Выноси логи (`/var/log`) и данные БД на разные физические диски.
- **RAID:** Переход с RAID 5 (медленная запись из-за подсчета четности) на RAID 10.
