# 1. Что такое eBPF? (Extended Berkeley Packet Filter)
Раньше (в классическом BPF) это была штука только для фильтрации сетевых пакетов (тот самый `tcpdump`). **Extended (eBPF)** превратил это в полноценную виртуальную машину внутри ядра Linux.

Это технология ядра Linux, позволяющая запускать безопасные программы в "песочнице" (sandboxed) внутри ядра без изменения его исходного кода или загрузки модулей. Она используется для высокопроизводительного мониторинга (observability), сетевой фильтрации, безопасности и профилирования, предоставляя возможности системного уровня.

**Суть:** Ты можешь запустить свой код прямо в ядре, не пересобирая его и не загружая опасные модули ядра (LKM).
- **Безопасность:** Перед запуском код проверяет **Verifier**. Если твой код может повесить ядро или залезть в чужую память — он просто не скомпилируется.
- **Скорость:** Код компилируется в машинный через **JIT (Just-In-Time)** компилятор и работает на скорости ядра.

# 2. Крючки (Probes)
Чтобы eBPF что-то считал, его нужно «приклеить» к какому-то событию в системе. Есть 4 основных типа:

1. **kprobes (Kernel Probes):** Динамическая вставка в любую функцию ядра. (Например, `vfs_read`).
2. **uprobes (User Probes):** То же самое, но для функций в пользовательских программах (например, приклеиться к функции в `nginx` или `mysql`).
3. **tracepoints:** Статические маркеры, которые разработчики ядра уже расставили в важных местах. Они стабильнее, чем kprobes (имена функций в ядре могут меняться от версии к версии, а tracepoints — нет).
4. **USDT (Userland Statically Defined Tracing):** Статические маркеры внутри приложений (например, в PostgreSQL).

# 3. bpftrace
Если **BCC** (другой инструментарий eBPF) требует писать на Python/C, то **bpftrace** — это высокоуровневый язык (похож на AWK), на котором можно писать однострочники прямо в консоли.
**Структура команды:**
`bpftrace -e 'probe /filter/ { action }'`

- **Probe:** Где ловим (например, `kprobe:vfs_read`).
- **Filter:** Какое условие (например, только для PID 123).
- **Action:** Что делаем (например, печатаем текст или инкрементируем счетчик).

**Пример (кто читает файлы прямо сейчас):**
```shell
bpftrace -e 'kprobe:vfs_read { @[comm] = count(); }'
```
*Здесь `@` — это ассоциативный массив (map), `comm` — встроенная переменная (имя процесса).*

# 4. execsnoop
Это одна из самых полезных утилит из набора `bcc-tools` (или написанная на `bpftrace`).

**Проблема:** Классический `top` или `ps` показывают процессы, которые живут долго. Если какой-то скрипт запускает сотни короткоживущих процессов (длительностью 0.1 сек), ты их не увидишь в `ps` — они исчезнут до того, как ты нажмешь Enter.

**Как работает `execsnoop`:** Он цепляется к системному вызову `execve()` (именно этот вызов отвечает за запуск новой программы). Каждый раз, когда в системе кто-то делает `exec()`, утилита выплевывает строку:
- **PID:** Кто запустился.
- **PPID:** Кто его «родил» (родительский процесс).
- **COMM:** Имя команды.
- **ARGS:** Все аргументы запуска (пароли в командной строке, пути и т.д.).

# 5. Другие команды
- **biolatency:** Показывает распределение задержек дискового ввода-вывода (I/O). Не просто «среднее», а гистограмму. Ты увидишь, что 90% запросов быстрые, а 10% — дико тормозят.
- **ext4slower (или xfs-slower):** Показывает только те операции чтения/записи, которые длились дольше, чем, например, 10мс. Помогает найти «тормозящий» файл.
- **tcptop:** Показывает пропускную способность сети по процессам в реальном времени, не нагружая систему так сильно, как `tcpdump`.
