# 1. Что это
LVM (**Logical Volume Manager**) — это слой абстракции между физическими дисками и файловой системой.
___
# 2. Архитектура LVM
- **PV (Physical Volume):** Физический уровень. Это твой диск или раздел (`/dev/sdb1`).
- **VG (Volume Group):** Уровень абстракции. Мы берем несколько PV и объединяем их в «виртуальный диск» — общую кучу места.
- **LV (Logical Volume):** Уровень «нарезки». Внутри VG мы создаем логические разделы, на которые уже накатываем файловую систему (ext4, xfs).
___
# 3. Изменение размеров «на лету»
Это главная фишка LVM. Ты можешь расширить диск, не отключая базу данных или веб-сервер.
## 3.1. Расширение
1. **Добавляем место в VG:** Если есть свободные PV, пропускаем. Если нет — втыкаем новый диск, делаем его PV и добавляем в группу: `pvcreate /dev/sdc` `vgextend my_vg /dev/sdc`
2. **Расширяем LV:** `lvextend -L +10G /dev/my_vg/mysql_lv` (добавить 10 ГБ) Или `lvextend -l +100%FREE /dev/my_vg/mysql_lv` (забрать всё свободное).
3. **Расширяем Файловую Систему (Критично!):** LVM расширил «железо», но ФС об этом не знает.
    - Для **ext4**: `resize2fs /dev/my_vg/mysql_lv`
    - Для **XFS**: `xfs_growfs /mnt/mysql` (у XFS указывается точка монтирования, а не устройство).
## 3.2. Уменьшение
- **XFS уменьшать нельзя.** Вообще. Только бэкап -> пересоздание -> восстановление.
- **ext4 можно**, но только **Offline** (сначала `umount`). _Порядок обратный:_ сначала уменьшаем ФС (`resize2fs`), потом LV (`lvreduce`). Если сделаешь наоборот — уничтожишь данные.
___
# 4. Снапшоты
В LVM снапшоты работают по принципу **COW (Copy-on-Write)**.
- Когда ты создаешь снапшот, он почти не занимает места.
- Как только данные в оригинальном LV меняются, старые блоки копируются в область снапшота.
- **Зачем это нужно:** Для консистентного бэкапа БД. Заморозил ФС -> сделал снапшот -> разморозил -> спокойно бэкапишь снапшот, пока база работает.
___
# 5. Восстановление и Troubleshooting
## 5.1. Потеря физического диска (PV)
Если один из дисков в VG "отвалился", VG перейдет в состояние `partial`.
- Посмотреть статус: `vgdisplay -v`
- Если диск умер окончательно и данных на нем не было (или был зеркальный LV), можно удалить «мертвый» PV: `vgreduce --removemissing my_vg`
## 5.2. Перенос данных между дисками (Pvmove)
Нужно заменить старый медленный HDD на новый SSD без остановки системы?
1. Добавляешь новый PV в текущую VG (`vgextend`).
2. Запускаешь магию: `pvmove /dev/sdb /dev/sdc` 
   _Ядро само будет перетаскивать блоки данных с одного диска на другой в фоновом режиме._
3. Удаляешь старый диск из VG (`vgreduce`).
## 5.3. Восстановление метаданных
LVM хранит бэкапы своей структуры в `/etc/lvm/archive`. Если ты случайно удалил LV, его можно вернуть:
1. `vgcfgrestore --list my_vg` (найти конфиг до удаления).
2. `vgcfgrestore -f /etc/lvm/archive/my_vg_XXXX.vg my_vg`.
