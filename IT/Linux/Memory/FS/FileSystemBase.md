# 1. База
В Linux все есть файл. Таким образом, получается, что файл является не единицей хранения информации, а единицей *обеспечения доступа* к информации.
В Linux файлы одновременно могут хранить информацию, выступать в роли «поставщиков» доступа к информации, хранимой на различных носителях, а также предоставлять доступ к клавиатуре и мыше.
О каждом файле хранятся метаданные. Номер можно увидеть, использовав команду `ls -la` , номер будет первым слева в выводе . Это номер [[Inodes|индексного дескриптора]]. Он содержит список стандартных свойств файла . Полный набор метаданных можно увидеть с помощью команды `stat <file>`.

# 2. Поддиректории root
- `/bin` (binary) - Содержит бинарные файлы основных системных утилит командной строки (`cat`, `ls`, etc.). В современных системах используется в виде symlink на `/usr/bin`.
- `/boot` - содержит основные файлы загрузки системы и само ядро Linux.
- `/dev` (devices) - содержит файлы устройств типо жестких дисков, принтеров, дисков, клавиатур и т.д. 
- `/etc` (etcetera) - содержит все общесистемные конфигурационные файлы. Содержит, например, настройки оконного менеджера, браузера и т.д.
- `/lib` - содержит основные настройки программ, лежащих в каталоге `/bin`. Как в случае с `/bin`, является символьной ссылкой на каталог `/usr/bin`.
- `/lib32` & `/lib64` - 32-ух и 64-битные библиотеки, соответственно. Если не установлены, например, 32-битные программы, то `/linb32` будет отсутствовать.
- `/media` - сюда подключаются съемные носители типо флешек. Также сюда принято монтировать дополнительные жесткие диски или удаленные файловые системы.
- `/mnt` (Mount) - ранее работала вместо `/media`.
- `/opt` (Optional) - содержит дополнительное ПО, как правило проприетарное, например, Google Chrome.
- `/proc` (Processes) - содержит виртуальную файловую системы `procfs`, которая предоставляет различную информацию из ядра + отображает системные процессы.
- `/run` - отображает информацию о запускаемых в процессы загрузки системы службах, демонах, процессах и т.д. Информация временная и удаляется при каждой перезагрузке.
- `/sbin` - аналог каталога `/bin`, однако содержит вспомогательные утилиты для администрирования и работы системы например `iptables`, `useradd` и т.д.
- `/srv` - предполагалось размещать сервисы, предоставляемые, например, веб-серверами, типо Nginx, однако фактически нигде не используется. 
- `/sys` - содержит различную информацию об устройствах, предоставляемую ядром Linux. Например, отсюда берется информация об оборотах вентиляторов, температуре процессора, информации о планировщике IO (ввода/выводы) жестких дисков и т.д. 
- `/tmp` (Temporary) - каталог для временных файлов, типо распаковки архивов при их открытии. Содержит распаковки пакетов во время обновления системы. Зачастую переносят на RAM через  `tmpfs`.
- `/var` -  содержит все системные журналы, логи, кэш пакетных менеджеров.
- `/usr` - фактически является вторичной иерархией файловой системы. Все программы, которые устанавливаются из репозиториев распаковываются в `/usr`, и уже в нем уже происходит разделение пакеты программы на составные части. 
   -  Исполняемые файлы идут в `/usr/bin`. 
   - Библиотеки в `/usr/lib`. 
   - Заголовочные файлы, требующиеся при разработке или сборке другой программы из исходного кода, если таковые имеются в `/usr/include`.
   - `/usr/src`, как правило используется для размещения исходного кода, к примеру содержит исходный код ядра Linux.
   - `/usr/share` - содержит архитектурно независимые файлы, вроде ярлыков приложений, тем и значков.
   - `/usr/local` - содержит третичную иерархию, куда производится установка локально собранных из исходного кода программ, если пользователь хочет сделать их общесистемными.
- `/home` содержит домашние каталоги пользователей, в каждом из которых содержатся личные данные и настройки каждого пользователя.
- `/root` - домашний каталог пользователя ***root***.

# 3. Файловый дескриптор
**Файловый дескриптор** – неотрицательное целое число, которое используется в интерфейсе между пространством пользователя и пространством ядра (kernel) для идентификации ресурсов файла / сокета. Когда создаётся новый поток ввода-вывода, ядро возвращает процессу, создавшему поток ввода-вывода, его файловый дескриптор.

Важно отметить, что файловые дескрипторы не ограничиваются только файлами.  
Они также могут ссылаться на каталоги, сокеты, каналы (pipes), устройства ввода-вывода и даже некоторые специфические для процесса ресурсы, такие как файлы процесса и области памяти.

Операционная система автоматически создает три файловых дескриптора для каждого процесса: 0 (стандартный ввод), 1 (стандартный вывод) и 2 (стандартный вывод ошибок).
Когда процесс делает `open()`:
```
процесс
   ↓
file descriptor (число)
   ↓
таблица открытых файлов ОС
   ↓
inode
   ↓
блоки данных на диске
```
## 3.1. Иерархия лимитов
В Linux существует три уровня ограничений на количество открытых файлов. Если «упрешься» хотя бы в один — получишь ошибку.
### 3.1.1. Системный лимит (System-wide)
Ограничивает суммарное количество открытых файлов для **всех** процессов в ОС.
- **Где смотреть:** `cat /proc/sys/fs/file-max`
- **Как менять:** Через `sysctl -w fs.file-max=XXXXX` (или в `/etc/sysctl.conf`).
### 3.1.2. Пользовательский лимит (ulimit / rlimit)
Ограничивает ресурсы для конкретного юзера. Делится на два типа:
- **Soft limit:** Тот, который юзер может поднять сам (в пределах Hard).
- **Hard limit:** Жесткий потолок, который может менять только root.
- **Где смотреть:** `ulimit -n` (покажет Soft). Чтобы увидеть всё: `ulimit -Sn` (soft), `ulimit -Hn` (hard).
- **Где менять:** В `/etc/security/limits.conf`.
### 3.1.3. Лимит конкретного процесса
Даже если ты поменял `ulimit` в конфигах, процесс, запущенный **до** изменений (например, через systemd), будет жить со старыми лимитами.
- **Где смотреть:** `cat /proc/[PID]/limits`. Ищи строку `Max open files`.
## 3.2. Поиск утечек через /proc/PID/fd
Если количество открытых дескрипторов постоянно растет и не падает — это утечка (программист забыл `close()`).
### 3.2.1. Поиск процесса
Быстрый способ увидеть, у кого больше всего открытых файлов:
```shell
lsof -n | awk '{print $2}' | sort | uniq -c | sort -nr | head -n 10
```
### 3.2.2. Инспекция через /proc
Директория `/proc/[PID]/fd/` содержит симлинки на все открытые процессом ресурсы.
```shell
ls -l /proc/[PID]/fd
```
Ты увидишь что-то вроде:
- `0 -> /dev/null` (stdin)
- `1 -> /var/log/app.log` (stdout)
- `2 -> /var/log/app.err` (stderr)
- `3 -> socket:[123456]` (сетевое соединение)
- `4 -> anon_inode:[eventpoll]` (внутренние механизмы ядра)
### 3.2.3. Группировка и анализ
Чтобы понять, **что именно** течет, сгруппируй типы файлов:
```shell
ls -l /proc/[PID]/fd | awk -F'-> ' '{print $2}' | cut -d: -f1 | sort | uniq -c | sort -nr
```
Если ты видишь 5000 строк `socket` или 5000 одинаковых путей к лог-файлу — ты нашел виновника.
