# 1. VIRT (Virtual Memory)
**VIRT** — это общий объем виртуального адресного пространства, который процесс «застолбил» за собой.

- **Что туда входит:** Весь исполняемый код, все данные, выделенные библиотеки, а также вся память, которую процесс запросил у ядра (через `malloc` или `mmap`), **но еще не обязательно начал использовать**.
- **Особенность:** VIRT может быть в разы больше, чем физическая оперативная память (RAM) на сервере. Это просто записи в таблице страниц, которые говорят: «Если процессу понадобится этот адрес, ядро что-нибудь придумает».
- **Зачем мониторить:** Если VIRT растет бесконечно, у приложения может быть утечка «адресного пространства» или оно очень неэффективно работает с памятью (например, мапит огромные файлы).

# 2. RES (Resident Memory)
**RES (Resident Set Size)** — это самое важное число. Это объем **физической RAM**, которую процесс занимает прямо сейчас.

- **Что туда входит:** Только те страницы памяти, к которым процесс реально обратился (записал туда что-то или прочитал).
- **Механизм (Page Fault):** Когда процесс обращается к адресу из VIRT, которого еще нет в RES, происходит «Page Fault» (страничный сбой). Ядро ловит это событие, находит свободный физический фрейм в RAM и отображает его на виртуальный адрес. Только в этот момент RES увеличивается.
- **Зачем мониторить:** Это основной показатель потребления ресурсов. Если сумма RES всех процессов приближается к объему RAM — жди OOM Killer.
___
# 3. SHR (Shared Memory)
**SHR (Shared Memory)** — это часть памяти процесса, которая потенциально может быть разделена с другими процессами.

- **Что туда входит:**
    1. **Общие библиотеки (Shared Libs):** Например, `libc.so` загружена в память один раз, но используется сотней процессов. Она будет в SHR у каждого.
    2. **Shared Memory Segments:** Специально созданные области для общения между процессами (Inter-Process Communication, IPC).
    3. **Бинарный код:** Сам исполняемый файл процесса часто помечается как shared.
- **Формула:** На самом деле, $RES = \text{Private Memory} + \text{Shared Memory}$.
- **Зачем мониторить:** Если ты видишь 10 воркеров Nginx, у каждого из которых RES по 100 МБ, а SHR по 80 МБ — это значит, что на самом деле они потребляют не $10 \times 100 = 1000$ МБ, а гораздо меньше, так как 80 МБ у них общие.

**Вопрос:** Если RES врет из-за SHR, как узнать, сколько памяти реально ест процесс? **Ответ:** Нужно смотреть **PSS (Proportional Set Size)**.
- PSS считает всю приватную память процесса + (Shared память / количество процессов, использующих её).
- PSS — самый честный показатель. Его можно найти в `/proc/[PID]/smaps`.
