# 1. Анатомия сокета в ядре
Внутри ядра Linux сокет представлен сложной иерархией структур данных. Основные из них:
1. **`struct socket` (BSD socket):** Верхний уровень. Это то, что видит VFS (Virtual File System). В этой структуре хранится состояние (соединен/нет) и указатель на файл.
2. **`struct sock` (INET socket):** Нижний уровень, специфичный для протокола. Здесь хранятся буферы приема/передачи, очереди прерываний, TTL, MSS и другие параметры протокола.
3. **Очереди (Queues):** У каждого сокета есть две основные очереди:
    - **Receive Queue (`sk_receive_queue`):** Сюда ядро складывает пришедшие и очищенные от заголовков данные, пока процесс не заберет их через `read()` или `recv()`.
    - **Write Queue (`sk_write_queue`):** Сюда процесс пишет данные, которые ядро затем упаковывает в пакеты и отправляет в сетевую карту.

# 2. Жизненный цикл TCP-сокета
Когда ты пишешь код на Go/Python/C, за кадром происходят следующие syscalls:

1. **`socket()`**: Ядро выделяет структуру в памяти и возвращает **File Descriptor (FD)**. На этом этапе сокет — это просто «пустая труба».
2. **`bind()`**: Сокету назначается локальный адрес и порт. Ядро проверяет, не занят ли порт другим процессом.
3. **`listen()`**: Сокет переходит в состояние ожидания. Ядро выделяет два бэклога (очереди):
    - **SYN Queue**: Для полуоткрытых соединений (пришел SYN, ждем ACK от клиента).    
    - **Accept Queue**: Для полностью установленных соединений (Handshake завершен). Лимит этой очереди — тот самый `net.core.somaxconn`.    
4. **`accept()`**: Процесс засыпает, пока в Accept Queue не появится соединение. Когда оно появляется, `accept()` вынимает его из очереди и создает **новый сокет** (с новым FD) специально для этого клиента. Оригинальный сокет продолжает «слушать».
5. **`connect()`**: Используется клиентом для инициации Handshake.

# 3. Unix Domain Sockets vs Network Sockets
## 3.1. Unix Domain Sockets (UDS / `AF_UNIX`)
- **Где:** Внутри одного хоста. Адрес — путь к файлу (напр. `/var/run/docker.sock`).
- **Как работают:** Данные копируются напрямую из адресного пространства одного процесса в буфер другого через ядро.
- **Профит:** Нет инкапсуляции, нет расчета контрольных сумм TCP/IP, нет фрагментации. Это **самый быстрый** способ передачи данных между процессами после Shared Memory.
## 3.2. Network Sockets (`AF_INET` / `AF_INET6`)
- **Где:** Внутри сети (или через `127.0.0.1`).
- **Как работают:** Проходят через весь сетевой стек: добавление заголовков TCP -> IP -> Ethernet (L4-L3-L2).
- **Проблема Loopback:** Даже если ты общаешься по `localhost`, ядро всё равно делает полную инкапсуляцию, хотя пакет и не покидает пределы процессора. Это медленнее, чем UDS.

# 4. Состояния сокетов и дебаг
Ты видишь их в выводе `ss -tapn` или `netstat`.
- **ESTABLISHED**: Соединение установлено, данные могут передаваться.
- **TIME_WAIT**: Сокет закрыт со стороны сервера, но ядро держит его в памяти еще некоторое время (обычно 60 сек), чтобы убедиться, что запоздавшие пакеты из сети не попадут в новое соединение с тем же портом.
    - _Проблема:_ Большое количество `TIME_WAIT` может забить таблицу сокетов и порты. Решение — `net.ipv4.tcp_tw_reuse`.
- **CLOSE_WAIT**: Приложение на другой стороне закрыло соединение, но **наше приложение** еще не вызвало `close()`.
    - _Кейс:_ Если видишь много `CLOSE_WAIT` — это **баг в коде** твоего приложения (утечка дескрипторов сокетов).

# 5. Буферы сокетов
В Linux пакеты представлены структурой `sk_buff`. Это самая важная структура для сетевой производительности. Когда данные приходят из сети:

1. Сетевуха по DMA (Direct Memory Access) пишет пакет в RAM.
2. Ядро создает `sk_buff`, который «оборачивает» эти данные.
3. По мере прохождения стека вверх (L2 -> L3 -> L4), ядро не копирует данные пакета, а просто **сдвигает указатели** в заголовке `sk_buff`. Это называется "Zero-copy" внутри ядра.
