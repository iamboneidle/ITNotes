# 1. Что это?
**ndots** (number of dots) — это пороговое значение количества точек в доменном имени, которое определяет, как резолвер (библиотека в ОС) будет искать адрес.

Логика проста:
- Если в запрашиваемом имени **меньше** точек, чем указано в `ndots`, то резолвер сначала попытается дополнить имя локальными поисковыми доменами из списка `search`.
- Если точек **столько же или больше**, резолвер сначала попробует разрешить имя как «абсолютное» (FQDN).
___
## 2. Как это работает на примере
Предположим, ваш `/etc/resolv.conf` выглядит так:
```
search dev.example.com example.com
nameserver 8.8.8.8
options ndots:5
```
Вы выполняете команду: `curl google.com`.
В имени `google.com` **одна** точка. А наш лимит `ndots` — **5**.
Так как $1 < 5$, Linux начнет перебирать суффиксы из списка `search`:
1. Запрос: `google.com.dev.example.com.` -> Ответ: NXDOMAIN (не найдено)
2. Запрос: `google.com.example.com.` -> Ответ: NXDOMAIN
3. Запрос: `google.com.` (только теперь как абсолютное имя) -> Ответ: OK!

**Итог:** Чтобы просто открыть Google, система сделала **3 DNS-запроса** вместо одного.

# 3. Проблема 'ndots:5' в Kubernetes
В Kubernetes по умолчанию установлено значение **`ndots:5`**. Это сделано для того, чтобы микросервисы могли легко общаться друг с другом по коротким именам (например, `db-service` или `db-service.staging.svc.cluster.local`).

Если сервис `A` хочет достучаться до сервиса `B` в том же неймспейсе, он просто пишет `http://b`. Точек ноль, $0 < 5$, и резолвер быстро находит его через `b.namespace.svc.cluster.local`.

**Но если ваше приложение лезет во внешний мир (в API соцсетей, базы данных вне кластера и т.д.):**
Для любого внешнего адреса (например, `api.stripe.com` — 2 точки) Kubernetes выполнит:
1. `api.stripe.com.default.svc.cluster.local` -> NXDOMAIN
2. `api.stripe.com.svc.cluster.local` -> NXDOMAIN
3. `api.stripe.com.cluster.local` -> NXDOMAIN
4. ...и так далее, пока не дойдет до внешнего резолвинга.

**Последствия:**
- **Latency:** Каждое внешнее соединение тратит время на 4–5 бесполезных DNS-запросов.
- **Нагрузка на CoreDNS:** Количество запросов к DNS-серверу растет кратно (в 5–10 раз). CoreDNS начинает потреблять много CPU и может начать дропать запросы по таймауту.

# 4. Как с этим бороться?
## 4.1. Точка в конце имени (FQDN)
Самый простой и «костыльный» способ. Если вы напишете в коде или конфиге адрес с точкой на конце: `curl google.com.` Резолвер поймет, что это **абсолютное имя**, и проигнорирует `ndots` и `search`, выполнив ровно 1 запрос.
## 4.2. Тюнинг манифеста в Kubernetes
Вы можете переопределить настройки DNS для конкретного пода (если это Deployment/StatefulSet):
```yaml
spec:
  dnsConfig:
    options:
      - name: ndots
        value: "1"
```
> [!caution] 
> Установив `ndots:1`, вы ускорите внешние запросы, но теперь для обращения к сервисам в других неймспейсах придется писать их полное имя (`service.namespace.svc.cluster.local`). 
## 4.3. NodeLocal DNSCache
Это специальный аддон для Kubernetes, который запускает агент DNS на каждой ноде. Он не уменьшает количество запросов, но делает их обработку локальной, убирая сетевые задержки до центрального CoreDNS и снижая нагрузку на таблицу `conntrack`.
