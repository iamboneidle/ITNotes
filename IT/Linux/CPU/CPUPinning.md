# 1. cgroups
**cgroups** — это механизм ядра Linux, который позволяет объединять процессы в иерархические группы и ограничивать для них ресурсы (CPU, Memory, I/O, Network).
___
# 2. CPU Pinning
По умолчанию планировщик задач Linux (CFS — Completely Fair Scheduler) сам решает, на каком ядре запустить процесс. Он может перекидывать поток с ядра №1 на ядро №8 (это называется **Task Migration**).

**Почему миграция — это плохо?**
- **L1/L2 Cache Misses:** У каждого ядра свой быстрый кэш. Когда поток переезжает, его данные остаются в кэше старого ядра. Новое ядро вынуждено лезть в медленную оперативку.
- **NUMA (Non-Uniform Memory Access):** В многопроцессорных системах память физически ближе к одному из процессоров. Если процесс переехал на CPU, который находится далеко от «его» плашки памяти, задержки вырастут в разы.

**CPU Pinning** через `cpuset` фиксирует процесс на конкретных ядрах. Процесс больше не «гуляет» по процессору.
___
# 3. CPU Isolation
Привязки (pinning) мало. Даже если ты привязал свой Nginx к ядру №1, ядро Linux всё равно может подселить туда другие фоновые задачи, прерывания или системные процессы.

**Isolation** — это когда мы говорим ядру: «Вообще не используй эти ядра для обычных задач. Оставь их пустыми».

**Как это делается:**
1. **Параметр загрузки ядра `isolcpus`:** В конфиге GRUB пишется `isolcpus=1,2,3`. При загрузке планировщик Linux просто «вычеркивает» эти ядра из своего общего списка.
2. **Результат:** На этих ядрах не будет работать ничего, кроме того, что ты **явно** туда привяжешь через cgroups или команду `taskset`.

# 4. Как работают cgroups в планировщике (CFS)
Планировщик Linux (CFS) оперирует двумя основными понятиями для ограничения ресурсов: **Shares** и **Quota**.
## 4.1. CPU Shares (Относительный вес)
Файл: `cpu.shares`.
- Это **мягкое** ограничение. Если система простаивает, процесс может забрать хоть 100% всех ядер.
- Но как только начинается борьба за ресурсы, планировщик делит время согласно «акциям» (shares). Если у Группы А — 1024 shares, а у Группы Б — 2048, то Б получит в два раза больше времени, чем А.
## 4.2. CPU Quota & Period (Жесткий лимит)
Файлы: `cpu.cfs_quota_us` и `cpu.cfs_period_us`.
- Это **жесткое** ограничение (Limit в Kubernetes).
- **Period:** Отрезок времени (например, 100мс).
- **Quota:** Сколько микросекунд внутри этого периода процессу разрешено работать.

_Пример:_ Если период 100мс, а квота 50мс — процесс получит только 50% мощности одного ядра и ни каплей больше, даже если остальные ядра свободны.

> **Troubleshooting Point (Throttling):** Если квота исчерпана до конца периода, процесс **замораживается** (throttling). В Kubernetes это главная причина тормозов (Latency spikes) при живом CPU. Проверяется в файле `cpu.stat` (параметр `nr_throttled`).

# 5. Инструментарий и дебаг
- **`taskset -p -c <PID>`**: Посмотреть или изменить привязку процесса к ядрам «на лету».
- **`numactl --hardware`**: Посмотреть топологию твоих процессоров и памяти (NUMA-узлы).
- **`/sys/fs/cgroup/cpu/...`**: Прямой доступ к параметрам групп.
- **`top` / `htop`**: Нажми `f` и добавь колонку **LAST CPU**, чтобы видеть, по каким ядрам прыгает твой процесс.
