# Вопросы
#### 1. Что такое индекс и что такое ключ?
>**Ключ** — это логическое понятие для идентификации данных, а **индекс** — это физическая структура для ускорения поиска.
>
>**Ключ**
>Это логическое ограничение (constraint), которое определяет правила для данных в таблице. Ключи нужны, чтобы поддерживать целостность данных.
>- Primary Key
>- Foreign Key
>- Unique Key
>
>**Индекс**
>Это скрытая служебная структура (аналог алфавитного указателя в конце книги), которая создается поверх таблицы, чтобы база данных могла находить строки быстрее, не сканируя всю таблицу целиком (**Full Table Scan**).
#### 2. Каковы преимущества и недостатки индексов?
>**Плюсы**
>- Колоссальное ускорение поиска (`SELECT`)
>- Оптимизация сортировки (`ORDER BY`)
>- Ускорение соединений (`JOIN`)
>- Обеспечение уникальности
>**Минусы**
>- Замедление записи (`INSERT`, `UPDATE`, `DELETE`)
>- Потребление дискового пространства
>- Нагрузка на RAM
>- Фрагментация
>  Со временем, при частом удалении и вставке данных, индексы становятся фрагментированными (появляются пустоты в страницах). Это требует периодического обслуживания (например, `REINDEX` в PostgreSQL или `OPTIMIZE TABLE` в MySQL).
#### 3. Представьте, что вы разрабатываете систему биллинга, которая должна обрабатывать тысячи счетов. Какую стратегию обновления данных вы бы выбрали?
>При обработке тысяч счетов я бы выбрал комбинированную стратегию, основанную на **асинхронной пакетной обработке** и **транзакционной целостности**.
>
>1. Очереди сообщений
>   Вместо того чтобы пытаться обновить тысячи строк одним тяжелым SQL-запросом, я бы использовал очередь (например, **RabbitMQ** или **Kafka**).
>2. Пакетная обработка
>   Обновлять записи по одной — слишком накладно для базы (много сетевых оверхедов и коммитов). Обновлять все 100 000 за раз — риск заблокировать таблицу.
>	- **Стратегия:** Использовать **Chunks** (пакеты) по 500–1000 записей.
#### 4. Какие методы чаще всего используют для масштабирования реляционных баз данных?
>1. **Вертикальное масштабирование**
>   Это самый простой путь: добавление ресурсов (CPU, RAM, Disk) на текущий сервер.
>2. **Репликация**
>   Разделение нагрузки по типу операций. Один сервер работает на запись (**Master/Primary**), а один или несколько — только на чтение (**Slaves/Replicas**).
>3. **Шардирование**
>   Это разделение данных одной таблицы на несколько разных серверов (шардов) по определенному признаку (например, `user_id`).
>	- **Как это работает:** Пользователи с ID 1–10000 живут на сервере А, а 10001–20000 — на сервере Б.
#### 5. Опишите механизм транзакций БД.
>**ACID**
>- A - Atomicity (Атомарность);
>- C - Consistency (Согласованность);
>- I - Isolation (Изоляция);
>- D - Durability (Долговечность).
>
>**Жизненный цикл транзакции**
>- **BEGIN** (или `START TRANSACTION`): Открывает блок транзакции.
>- **COMMIT**: Фиксирует изменения. После этого данные видны всем и записаны на диск.
>- **ROLLBACK**: Отменяет все изменения, сделанные внутри текущей транзакции.
>  
>**WAL (Write-Ahead Logging)**
>- Сначала изменения записываются в специальный лог-файл — **WAL**. Эта запись очень быстрая, так как идет в конец файла.
>- Только после того, как запись в WAL подтверждена диском, транзакция считается завершенной.
>**MVCC (Multi-Version Concurrency Control)**
>- Когда транзакция А меняет строку, транзакция Б все еще видит «старую версию» этой строки.
>- Это позволяет читать данные без блокировок (Lock-free), даже если в этот момент идет активная запись.
>  
>**Уровни изоляции (Isolation Levels)**
>1. **Read Uncommitted:** Можно видеть незавершенные изменения других транзакций («грязное чтение»).
>2. **Read Committed:** (Стандарт в Postgres/Oracle) Видны только зафиксированные изменения.
>3. **Repeatable Read:** (Стандарт в MySQL) Гарантирует, что повторное чтение данных в одной транзакции даст тот же результат.
>4. **Serializable:** Самый строгий уровень. Полная имитация последовательной работы.
#### 6. Как мы можем удалить таблицу или базу данных?
>Удаление таблицы (DROP TABLE) — `DROP TABLE users;`
>Удаление базы данных (DROP DATABASE) — `DROP DATABASE production_db;`
>
>- **DROP** — Удаляет таблицу целиком из схемы. Отката нет (в большинстве БД). Делает мгновенно
>- **TRUNCATE** — Удаляет все данные, но оставляет структуру. Откат зависит от БД (в PG — да, в MySQL — нет). Делает очень быстро
>- **DELETE** — Удаляет строки по одной (можно с `WHERE`). Да. Медленно
#### 7. Как найти медленные запросы в MySQL/PostgreSQL?
>- Механизм Slow Query Log в MySQL.
>- Модуль pg_stat_statements
>  В Postgres лучший способ — использование расширения `pg_stat_statements`. Оно собирает статистику по всем запросам (количество вызовов, общее время, нагрузка на диск).
>- Анализ конкретного запроса (EXPLAIN)
>  Когда вы нашли «виновника», нужно понять, **почему** он медленный. Для этого используется команда `EXPLAIN`.
#### 8. Какие SQL-операторы манипулирования данными вы знаете?
>- **DDL — Data Definition Language**
>	- *CREATE*: Создание объектов
>	- *ALTER*: Изменение структуры существующего объекта
>	- *DROP*: Полное удаление объекта
>	- *TRUNCATE*: Быстрая очистка всех данных из таблицы
>- **DML — Data Manipulation Language**
>	- *SELECT*: Чтение данных
>	- *INSERT*: Добавление новых строк
>	- *UPDATE*: Изменение существующих записей
>	- *DELETE*: Удаление строк
>- **DCL — Data Control Language**
>	- *GRANT*: Предоставление пользователю прав на определенные действия
>	- *REVOKE*: Отзыв ранее выданных прав
>- **TCL — Transaction Control Language**
>	- *COMMIT*: Сохранение всех изменений, сделанных в текущей транзакции
>	- *ROLLBACK*: Отмена изменений и возврат к состоянию до начала транзакции
>	- *SAVEPOINT*: Создание промежуточной точки внутри транзакции, к которой можно откатиться.
#### 9. Можно ли вывести список баз данных/таблиц через CLI? Как мы можем переключаться между базами данных MySQL/PostgreSQL?
>MySQL CLI (`mysql`)
>Для работы вам нужно сначала подключиться к серверу: `mysql -u username -p`.
>
>PostgreSQL использует «мета-команды», которые начинаются с обратного слэша. Они работают быстрее, чем стандартные SQL-запросы к системным таблицам.
>Подключение: `psql -U username -d database_name`
#### 10. Какие storage engines в MySQL вы знаете? Какие отличия?
>В MySQL есть два основных движка: **InnoDB** и **MyISAM**.
>1. **InnoDB** — современный стандарт, который поддерживает транзакции (ACID), внешние ключи и блокировки на уровне строки, что делает его идеальным для высокой нагрузки.
>2. **MyISAM** — устаревший движок без поддержки транзакций, использующий блокировку всей таблицы, но иногда более быстрый на простых операциях чтения.
>3. Разница критична для надежности: InnoDB восстанавливается после сбоев автоматически, а MyISAM легко повреждается при внезапной остановке сервера.
#### 11. Как работает репликация MySQL/PostgreSQL? Какие параметры должны быть настроены для репликации?
>- **MySQL:** Использует **Binary Log (binlog)**. Master записывает все изменения (запросы или изменения строк) в бинарный лог. Реплика подключается к Master, копирует записи в свой **Relay Log** и последовательно выполняет их.
>- **PostgreSQL:** Использует **Write-Ahead Log (WAL)**. Primary передает поток WAL-записей через протокол потоковой репликации (**Streaming Replication**). Реплика находится в режиме постоянного восстановления (Recovery), накатывая эти записи в реальном времени.
>  
>  **Ключевые параметры для настройки**
>- Для MySQL (`my.cnf`)
>	- `server_id`: Уникальный ID сервера в кластере.
>	- `log_bin`: Путь к бинарному логу (включает логирование на Master).
>	- `binlog_format`: Формат записи (`ROW` — самый надежный, `STATEMENT` или `MIXED`).
>	- `gtid_mode=ON`: Включает глобальные идентификаторы транзакций (упрощает управление репликацией).
>- Для PostgreSQL (`postgresql.conf` и `pg_hba.conf`)
>	- `wal_level`: Должен быть `replica` или выше.
>	- `max_wal_senders`: Количество одновременно подключенных реплик.
>	- `hot_standby = on`: Позволяет выполнять SELECT-запросы на реплике.
>	- `pg_hba.conf`: Нужно разрешить подключение пользователю с ролью `REPLICATION` с IP-адреса реплики.
#### 12. Сравните SQL и NoSQL.

|**Характеристика**|**SQL (Relational)**|**NoSQL (Non-relational)**|
|---|---|---|
|**Структура**|Строго определенная схема (таблицы).|Гибкая схема (документы, графы, пары ключ-значение).|
|**Масштабирование**|Преимущественно **вертикальное** (мощнее сервер).|**Горизонтальное** (добавление новых серверов).|
|**Целостность**|Соответствие **ACID** (строгая надежность).|Соответствие **BASE** (акцент на доступность).|
|**Язык запросов**|Стандартизированный **SQL**.|Разный для каждой БД (Object-oriented API, JSON).|
|**Примеры**|PostgreSQL, MySQL, Oracle, MS SQL.|MongoDB, Redis, Cassandra, Neo4j.|
#### 13. Sharding vs replication?
>**Replication** (Репликация) копирует данные для надежности и чтения, а **Sharding** (Шардирование) разделяет данные для масштабирования записи и объема.
>
>**Replication:**
>- **Как работает:** У вас есть один основной сервер (**Master/Primary**), куда идут все записи, и его копии (**Replicas/Slaves**), которые синхронизируются с ним.
>- **Зачем это нужно:**
>	- **Отказоустойчивость:** Если Master «умрет», одна из реплик станет новым Master.
>	- **Масштабирование чтения:** Вы можете распределить `SELECT` запросы между всеми репликами.
>- **Ограничение:** Не помогает, если данных слишком много для одного диска или если Master не справляется с потоком записей (`INSERT/UPDATE`).
>  
>**Sharding:**
>- **Как работает:** Каждая строка таблицы попадает на конкретный сервер (шард) на основе ключа шардирования (например, `user_id`). Пользователи 1–5000 — на Сервере А, 5001–10000 — на Сервере Б.
>- **Зачем это нужно:**
>	- **Объем:** Если база весит 10 ТБ, а диск сервера всего 2 ТБ.
>	- **Масштабирование записи:** Нагрузка на запись распределяется между всеми серверами.
>- **Ограничение:** Это очень сложно. Запросы, затрагивающие данные на разных шардах (Cross-shard JOINs), работают крайне медленно или требуют сложной логики в приложении.
#### 14. Какие есть виды индексов? Когда и зачем использовать?
>**B-Tree (Сбалансированное дерево)**
>- **Как работает:** Данные организованы в виде древовидной структуры, где каждый узел содержит отсортированные ключи и ссылки на дочерние узлы.
>- **Когда использовать:** Для операций сравнения (`=`, `>`, `<`, `>=`) и поиска по диапазону.
>- **Зачем:** Обеспечивает предсказуемое время поиска $O(\log n)$.
>
>**Hash-индексы**
>- **Как работает:** На основе значения ключа вычисляется хэш-код, который указывает на конкретную ячейку в таблице.
>- **Когда использовать:** Только для точных совпадений (оператор `=`).
>- **Зачем:** Работает быстрее B-Tree ($O(1)$), но бесполезен для поиска по диапазону или сортировки.
>
>**GIN (Generalized Inverted Index)**
>- **Как работает:** Содержит список значений и ссылается на все записи, где это значение встречается.
>- **Когда использовать:** Для поиска по массивам, JSONB и полнотекстового поиска.
>- **Зачем:** Чтобы быстро находить документы, содержащие определенный тег или ключ внутри JSON.
>
>**GiST (Generalized Search Tree)**
>- **Как работает:** Иерархический индекс, который позволяет индексировать сложные типы данных.
>- **Когда использовать:** Геоданные (координаты, полигоны), интервалы времени и полнотекстовый поиск.
#### 15. Требования к схеме БД. Character sets, collations, default, not null и т. д.
>**Кодировки и сопоставления (Character Sets & Collations)**
>- *Character Set (Кодировка):* Набор символов и их двоичное представление.
>- *Collation (Сопоставление):* Набор правил для сравнения и сортировки строк.
>
>**Ограничения целостности (Constraints)**
>- *NOT NULL:* Запрещает хранение «пустоты» (NULL).
>- *DEFAULT:* Значение по умолчанию.
>- *UNIQUE:* Гарантирует, что значения в колонке не повторяются.
>- *PRIMARY KEY (PK):* Уникальный идентификатор строки.
>- *FOREIGN KEY (FK):* Связывает таблицы между собой. Гарантирует, что вы не удалите категорию, пока в ней есть товары.
#### 16. Зачем и как тестировать перформанс баз данных?
>**Зачем**
>- *Определение предела (Capacity Planning):* Понять, сколько одновременных пользователей или транзакций (TPS) выдержит текущее железо до деградации.
>- *Поиск регрессий:* Проверить, не замедлил ли новый индекс или изменение схемы общую скорость работы.
>- *Оптимизация конфигурации*
>- *Стресс-тестирование:* Узнать, как база восстанавливается после критического пика или отказа одного из узлов репликации.
>
>**Как**
>- *Load Testing (Тест нагрузки):* Проверка работы при ожидаемом количестве запросов.
>- *Stress Testing (Стресс-тест):* Постепенное увеличение нагрузки до тех пор, пока база не начнет выдавать ошибки или время ответа (latency) не станет неприемлемым.
>- *Soak/Endurance Testing (Тест на выносливость):* Длительная нагрузка (часы или дни) для поиска утечек памяти или проблем с переполнением дискового пространства/логов.
