# Вопросы
#### 1. Что такое IaaC и зачем он нужен?
> Подход к управлению и настройке ИТ-инфраструктуры с помощью **файлов конфигурации** (кода), а не через ручную настройку серверов.
>- Автоматизация и скорость
>- Масштабируемость
>- Версионность
>- Снижение рисков: Минимизируется человеческий фактор
#### 2. Что такое Terraform?
> **Terraform** — это Open Source инструмент для **IaC (Infrastructure as Code)** от компании HashiCorp, который позволяет создавать, изменять и версионировать инфраструктуру безопасно и эффективно.
> 
> Основные фишки:
> - **Декларативный подход:** Ты описываешь в конфигах (`.tf`), **что** ты хочешь получить (например, «нужно 3 сервера»), а Terraform сам вычисляет, **как** это сделать.
> - **Провайдеры:** Terraform умеет работать практически с чем угодно через API — от облаков (AWS, Azure, GCP) до локальных решений (VMware, OpenStack) и даже SaaS (Cloudflare, Datadog).
> - **State File (`terraform.tfstate`):** Инструмент хранит текущее состояние инфраструктуры в специальном файле. Это позволяет ему понимать, какие ресурсы уже созданы, а какие нужно обновить или удалить.
> - **Execution Plan:** Перед применением изменений команда `terraform plan` показывает точный список действий, которые будут выполнены.
В отличие от того же Ansible (который лучше подходит для настройки софта _внутри_ ОС), Terraform идеален для создания **фундамента**: сетей, виртуалок, балансировщиков и баз данных.

#### 4. Структура Terraform. Как организовать multi-environment project? Terraform workspaces?
>Организация проекта в Terraform для нескольких окружений (**Dev, Staging, Prod**) — это классическая задача, которую можно решить двумя основными способами: через **структуру директорий** или через **Workspaces**. Выбор зависит от того, насколько сильно ваши окружения отличаются друг от друга.
>
>1. **Разделение по директориям**
>   Это самый надежный и популярный в индустрии подход. Каждое окружение — это отдельный конфигурационный сет со своим состоянием (State).

```
.
├── modules/               # Общие переиспользуемые модули
│   ├── vpc/
│   ├── db/
│   └── app/
└── environments/          # Специфичные для окружений папки
	├── dev/
	│   ├── main.tf        # Вызывает модули с параметрами для Dev
	│   ├── variables.tf
	│   └── backend.tf     # Хранит state в dev-папке (напр. S3 bucket/key)
	└── prod/
		├── main.tf        # Те же модули, но другие инстансы/лимиты
		├── variables.tf
		└── backend.tf
```
>Почему это хорошо:
>- **Изоляция:** Ошибка в коде для Dev никак не аффектит Prod State.
>- **Разные Backend:** Можно хранить стейт прода в более защищенном месте.
>- **Читаемость:** Вы четко видите, что развернуто в конкретной папке.
>- **Версионность:** Можно использовать разные версии модулей для разных сред.
>
>2. **Terraform Workspaces**
>   Workspaces позволяют использовать **один и тот же код** для управления разными состояниями. По умолчанию вы находитесь в воркспейсе `default`.
>   Вы создаете именованные пространства:
>   ```
terraform workspace new dev
terraform workspace new prod
>   ```
>   В коде вы можете обращаться к текущему окружению через переменную `${terraform.workspace}`.
>   Плюсы и минусы:
>   - **Плюс:** Минимум дублирования кода.
>   - **Минус:** Огромный риск. Легко забыть переключить workspace (`terraform workspace select prod`) и случайно снести продакшен, думая, что вы в деве.
>   - **Минус:** Нельзя настроить разные бэкенды или разные права доступа (RBAC) на уровне файлов.
#### 5. Лучшие практики по использованию многих Terraform states.
>1. **Декомпозиция по слоям (Layering)**
>   Разделяйте инфраструктуру на логические слои в зависимости от их жизненного цикла и частоты изменений.
>2. **Использование Remote State и Locking**
>   Никогда не храните state на локальной машине.
>3. Передача данных между States: `terraform_remote_state` vs SSM
>	1. **Data Source `terraform_remote_state`:** Позволяет одному проекту прочитать `outputs` другого проекта напрямую из его state-файла.
>	2. **SSM Parameter Store / Key Vault (Рекомендуется):** Первый проект записывает ID ресурса в хранилище секретов/параметров облака, а второй — читает его оттуда.
>4. **CI/CD как единственный источник изменений**
>   При использовании многих стейтов ручной запуск становится опасным.
#### 6. Что такое Terraform provider, module?
>**Provider (Провайдер)** — это плагин, который позволяет Terraform взаимодействовать с облачными провайдерами, SaaS-сервисами или локальными API.
>Terraform сам по себе не «знает», как создать виртуальную машину в AWS или базу данных в Azure. Он делегирует эту задачу провайдеру.
>
>**Module (Модуль)** — это набор конфигурационных файлов Terraform (`.tf`), сгруппированных в одной директории. Это основной инструмент для переиспользования и структурирования кода.
>Если провайдер — это библиотека функций, то модуль — это написанная вами (или сообществом) функция, которая создает целую пачку ресурсов разом.

#### 8. Когда нужно использовать local-exec и remote-exec?
>В Terraform использование `local-exec` и `remote-exec` считается **крайней мерой**. Эти инструменты (provisioners) используются только тогда, когда API облачного провайдера или существующие провайдеры Terraform не позволяют реализовать нужную логику.
>Основное правило: если можно сделать через `user_data`, `cloud-init` или нативный провайдер (например, `kubernetes` или `postgresql`), **используйте их**.
>
>1. **local-exec**
>   - Регистрация во внешних системах
>   - Запуск Ansible
>   - Подготовка локальных файлов
>2. remote-exec
>   - Финальная настройка ОС
>   - Установка софта «на лету»
#### 9. Что такое golden image и как его создать?
>**Golden Image** (также известный как «золотой образ», Master Image или Base Image) — это эталонный снимок настроенной операционной системы с уже установленным программным обеспечением, драйверами и конфигурациями безопасности.
>
>Современный стандарт автоматизации этого процесса — использование **HashiCorp Packer**. Он позволяет превратить описание образа в код.
#### 10. В чем главное отличие между Ansible and Terraform?
>Главное отличие — в их предназначении: **Terraform** создает инфраструктуру, а **Ansible** её настраивает.
>Teffaform — декларативный, Ansible — процедурный.
#### 11. Что такое state в контексте использования Terraform?
>В контексте Terraform **State** (состояние) — это критически важный файл (обычно `terraform.tfstate`), который служит «единым источником истины» для вашей инфраструктуры.
>
>Terraform — это декларативный инструмент. Вы описываете _желаемое_ состояние (например, «мне нужно 3 сервера»), а Terraform должен понять, что делать. Без файла State он бы не справился.
