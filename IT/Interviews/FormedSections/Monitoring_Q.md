# Вопросы
#### 1. Какие метрики нужно собирать? Разница между infrastructure и application monitoring.
>Инфраструктурный мониторинг следит за состоянием ресурсов хоста (CPU, RAM, Disk), используя метод **USE** (Utilization, Saturation, Errors). Прикладной мониторинг (APM) следит за качеством работы кода (RPS, ошибки, Latency), используя метод **RED** (Rate, Errors, Duration). Важно иметь оба: инфраструктура говорит, _на чем_ мы работаем, а приложение — _как_ мы обслуживаем пользователя.
#### 2. Какая разница между pull и push model в системах мониторинга?
>В **Pull-модели** сервер сам опрашивает цели по расписанию (как Prometheus). Это дает контроль над нагрузкой, но требует прямой видимости клиентов. В **Push-модели** клиенты сами шлют данные (как Zabbix). Это удобно для работы за NAT и мониторинга коротких скриптов, но сложнее контролировать поток входящих данных на сервере.
#### 3. Какая разница между Black box и White box monitoring?
>**Black Box** мониторинг проверяет внешние симптомы (доступность, время ответа) без доступа к коду. White Box собирает внутренние метрики системы (CPU, логи, состояние БД). Black Box нужен для оповещения о сбоях, а **White Box** — для поиска их причин.
#### 4. Расскажите о подходах к сбору application логов.
>1. **Запись в STDOUT/STDERR**
>   Это стандарт для контейнеризированных приложений (Docker, Kubernetes).
>2. **Запись в файл на диск**
>   Классический подход для монолитов или legacy-систем.
>3. **Отправка напрямую в хранилище (через SDK/Библиотеки)**
>   Приложение само отправляет логи по сети (HTTP, UDP, TCP).
>4. **Использование Sidecar-контейнера**
>Best practicies:
>- Всегда используйте **JSON**. Это позволяет легко искать по полям (например, по `user_id` или `error_code`), а не парсить текст регулярками.
>- Добавляйте уникальный ID к каждому запросу. Это позволит отследить путь одного запроса через цепочку из 10 микросервисов.
>- Писать структурированный JSON в STDOUT и использовать стек **Loki** или **Elasticsearch** для централизации».
