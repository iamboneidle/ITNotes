# Вопросы
#### 1. Что такое Continuous Integration и Continuous Deployment? В чем разница между Continuous Deployment и Continuous Delivery?
> **Continuous Integration (CI) — Непрерывная интеграция**
> **Суть:** Как только разработчик пушит код, автоматически запускается пайплайн: сборка приложения и прогон тестов
> **Continuous Delivery (CD) — Непрерывная доставка**
> **Ключевой момент:** Выпуск в продакшен происходит **вручную**. Код всегда готов к релизу, но решение принимает человек.
> **Continuous Deployment (CD) — Непрерывное развертывание**
> **Суть:** Каждый коммит, успешно прошедший через все этапы CI и тесты, **автоматически** улетает сразу в **Production** без участия человека.
#### 2. Расскажите о разновидностях тестов, которые мы можем использовать в CI пайплайне.
>- Static Analysis (Линтеры)
>- Unit Tests
>- Integration Tests
>- Smoke Tests
#### 3. Какие виды тестов вы знаете и зачем они нужны?
>- **Unit Tests**
>  Работа отдельного минимального куска кода (функции, метода, класса) в полной изоляции.
>- **Integration Tests**
>  Взаимодействие между двумя и более модулями или сервисами.
>- **End-to-End Tests**
>  Полный путь пользователя в реальной системе.
>- **Smoke Tests**
>  Самый критичный функционал после деплоя.
>- **Regression Tests**
>  Весь старый функционал после добавления новых фич.
>- **Load / Performance Tests**
>  Поведение системы под высоким трафиком.
#### 4. Deployment strategies. Какие существуют и чем отличаются (recreate, blue-green, canary etc.)?
>**Recreate (Пересоздание):**
>Самый простой и радикальный метод.
>- **Как работает:** Сначала полностью удаляются все старые поды (v1), и только после их удаления запускаются новые (v2).
>- **Downtime:** Есть. Приложение будет недоступно в промежутке между удалением старых и запуском новых контейнеров.
>
>**Rolling Update (Постепенное обновление)**
>Это стратегия **по умолчанию** в Kubernetes.
>- **Как работает:** Поды обновляются по очереди. K8s запускает один новый под, ждет его готовности (Readiness Probe), затем удаляет один старый.
>- **Downtime:** Нет. Трафик плавно перетекает со старых версий на новые.
>- **Настройка:** Регулируется параметрами `maxSurge` (сколько подов можно создать сверх нормы) и `maxUnavailable` (сколько подов можно одновременно «положить»
>- **Минусы:** В процессе деплоя в кластере одновременно живут v1 и v2. Код должен быть обратно совместим.
>
>**Blue-Green**
>- **Как работает:**
> 	 1. Рядом с работающей версией (Blue) разворачивается **полная копия** новой версии (Green).
> 	 2. Вы тестируете Green в изоляции.
> 	 3. Когда всё готово, вы меняете селектор в **Service** или **Ingress**, и весь трафик мгновенно уходит на Green.
>  - **Downtime:** Практически нет.
>  - **Плюсы:** Мгновенный откат (rollback) — просто переключите сервис обратно на Blue.
>  - **Минусы:** Дорого (нужно двойное количество ресурсов в кластере).
>
>**Canary (Канареечный деплой)**
>Метод тестирования на «живых» пользователях.
>- **Как работает:** Вы запускаете небольшую часть новых подов (например, 10%). Трафик распределяется между ними.
>- **Плюсы:** Минимизация рисков. Если в v2 есть баг, его увидит лишь малая часть пользователей.
>- **Реализация:** В чистом K8s это делается через манипуляции с метками (labels). В более продвинутых вариантах — через **Service Mesh (Istio)** или **Ingress-контроллеры** (например, Nginx Ingress позволяет делить трафик по заголовкам или кукам).
#### 5. Какие существуют branching strategy? На что опираться при выборе?
>1. **GitHub Flow (Short-lived branches)**
>   Самая популярная и простая стратегия для веб-сервисов и облачных приложений.
>	- *Принцип:* Есть только одна главная ветка (`main`). Для любой задачи создается короткоживущая ветка (`feature-branch`), которая после код-ревью и тестов сразу вливается в `main` и деплоится.
>	- *Плюс:* Идеально для CD (Continuous Delivery).
>	- *Минус:* Требует очень высокого покрытия тестами, так как `main` всегда должен быть готов к деплою.
>2. **GitFlow (Classic/Heavy)**
>   Строгая и сложная стратегия, которая была стандартом долгое время.
>	- *Принцип:* Использует две долгоживущие ветки (`main` для релизов и `develop` для интеграции), а также вспомогательные: `feature`, `release` и `hotfix`.
>	- *Плюс:* Отлично подходит для проектов с четким циклом релизов (например, мобильные приложения или коробочный софт).
>	- *Минус:* Огромное количество конфликтов при слиянии и сложность поддержки. Сегодня считается избыточной для большинства веб-проектов.
>3. **Trunk-Based Development (TBD)**
>	- *Принцип:* Разработчики вливают маленькие изменения в `main` (trunk) несколько раз в день. Никаких веток на неделю. Если фича не готова, она прячется за *Feature Flags*.
>	- *Плюс:* Минимальный "ад слияния" (merge hell) и самая быстрая обратная связь.
>	- *Минус:* Требует высочайшей квалификации и продвинутой системы CI.
>4. **GitLab Flow (Environment-based)**
>	- *Принцип:* К основной ветке `main` добавляются ветки окружений (например, `pre-production`, `production`). Код «протекает» сверху вниз по мере готовности.
>	- *Плюс:* Удобно, если инфраструктура жестко разделена на стейджи.
