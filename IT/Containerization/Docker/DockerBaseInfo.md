# 1. Инструменты в основе
Docker базируется на технологиях **namespaces**, **cgroups**, **capabilities**, **overlay**  
**namespaces** - обеспечивает изоляцию. Используется для изоляции. Например, можно айдишники процессов разместить в разных контейнерах.  
**cgroup** - штука, которая позволяет управлять группой процессов, и управлять их ресурсами.  
**capabilites** - штука, которая позволяет дать некоторые рут привелегии процессам или исполняемым файлам. Например, изменить UID процесса на 0, или дать возможность монтировать файловые системы.  
**overlay (overlayFS, overlay2-драйвер)** - файловая система, которая умеет работать "слоями". Не сохранять каждый раз новые файлы, а наслаивать один слой на другой, тем самым экономя место на диске и время создания контейнера.

А вот докер - это уже штука, которая всеми этими технологиями рулит. Да ещё и удобным для нас образом.

# 2. Компоненты
Компоненты докера:
1. Docker Daemon — то самое Container Engine; запускает контейнеры.
2. Docker CII — утилита по управлению Docker.
3. Dockerfile — инструкция по тому, как собирать образ.
4. Image — образ, из которого раскатывается контейнер.
5. Container.
6. Docker registry — хранилище образов.

# 3. Архитетура
Когда ты пишешь `docker run`, происходит цепочка вызовов через несколько слоёв. Снизу вверх:
- **Ядро Linux** предоставляет два механизма, на которых всё держится: **namespaces** (изоляция — процессы, сеть, файловая система) и **cgroups** (лимиты — CPU, память, I/O). Docker сам по себе никакой магии не делает — он просто удобно управляет этими встроенными возможностями ядра. [[ContainerDIY#1. Теория|см. подробнее тут]].
- **runc** — самый нижний уровень. Это маленькая утилита, которая непосредственно создаёт контейнер: вызывает ядро, настраивает namespaces и cgroups, запускает процесс. Больше ничего не умеет. Стандарт OCI — значит, другие системы (podman, kubernetes) используют тот же runc.
- **containerd** — уровень выше. Это полноценный демон, который управляет жизненным циклом контейнеров: скачивает образы, хранит их, запускает и останавливает контейнеры (делегируя **runc**), управляет сетью и volumes. Kubernetes, кстати, общается напрямую с containerd, минуя Docker.
- **dockerd (Docker daemon)** — верхний уровень. Принимает команды от CLI через REST API, реализует удобные абстракции (Dockerfile, docker-compose, networks, volumes как концепции), и делегирует реальную работу containerd. Исторически dockerd делал всё сам, но со временем внутренности выделили в отдельные компоненты.
- **docker CLI** — просто клиент. Отправляет HTTP-запросы к dockerd через unix-сокет `/var/run/docker.sock`. Сам ничего не делает.

Цепочка при `docker run` выглядит так:
```
docker CLI
    → dockerd  (REST API через /var/run/docker.sock)
        → containerd  (gRPC)
            → containerd-shim  (посредник, держит контейнер живым)
                → runc  (создаёт контейнер и завершается)
                    → ядро Linux (namespaces + cgroups)
```
`containerd-shim` — интересная деталь: runc после запуска контейнера завершает работу, а shim остаётся висеть как родительский процесс контейнера. Это позволяет перезапускать dockerd без убийства всех контейнеров.
